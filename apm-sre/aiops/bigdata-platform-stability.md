## 为什么需要掌握

- **岗位是「大数据平台」智能运维**：JD 第 1、3、4 条都围绕超大规模大数据平台的稳定性。如果只会讲 K8s 和微服务，会被认为「不懂被运维对象」。
- **AIOps Agent 必须懂被诊断对象的故障模式**：根因定位 Agent 的工具取证、知识库、SOP，本质都建立在「这个组件会怎么坏」之上。不懂故障模式，AIOps 就是空壳。
- **和我相邻经验高度相关**：调度积压、Spark on K8s、变更管控这些我真做过，组件故障模式是它的延伸，连接自然。
- **高频追问区**：面试官从 Agent 往下问一两层，必然落到具体组件。这是最容易暴露「只会概念」的地方。

## 它解决什么问题

大数据平台的稳定性问题，本质是几类反复出现的故障模式。理解它们，才能设计监控指标、告警规则、根因取证和 SOP。

- **流处理作业吞吐跟不上输入速率**
  - **对应故障模式**：Flink 反压（Backpressure）、Checkpoint 超时 / 失败、状态后端膨胀。
  - **面试表达**：反压不是病因而是症状，要顺着算子链往下游找瓶颈点。

- **消息消费跟不上生产，端到端延迟拉大**
  - **对应故障模式**：Kafka Consumer Lag 积压、频繁 Rebalance、ISR 收缩 / Under-replicated。
  - **面试表达**：积压先分清是「消费能力不足」还是「消费停了」，两者根因完全不同。

- **批处理作业变慢或失败，下游产出延迟**
  - **对应故障模式**：Spark 数据倾斜、Shuffle 溢写 / OOM、小文件过多、Executor 丢失。
  - **面试表达**：批处理变慢九成是倾斜和 Shuffle，先看 Stage 级别的 task 时间分布。

- **存储层抖动拖垮整个平台**
  - **对应故障模式**：HDFS NameNode 压力 / GC、Block 丢失 / 副本不足、DataNode 慢盘、小文件导致元数据爆炸。
  - **面试表达**：HDFS 是很多组件的共同底座，它一抖，上面的 Spark / Hive 全受影响，这是典型的「公共依赖放大故障」。

- **调度层任务堆积，产出 SLA 破线**
  - **对应故障模式**：调度实例积压、依赖卡死、资源池打满、单点慢任务阻塞队列、变更引入的批量失败。
  - **面试表达**：这是我真实做过的——调度积压要分清是上游依赖没就绪、资源不够，还是 Worker 执行慢。

- **元数据 / 中心服务成为隐藏单点**
  - **对应故障模式**：Hive Metastore 慢查询 / 连接打满、Schema 变更不兼容、分区元数据膨胀。
  - **面试表达**：Metastore 是隐藏的强依赖，它慢了表现为「所有 SQL 都卡在编译期」，很容易误判成引擎问题。

## 核心概念

只讲面试需要的故障模式，每个给出：定义、典型根因、关键指标、和我经验的映射、易被追问点。

### Flink 反压（Backpressure）

- **一句话**：下游算子处理速度跟不上上游，压力沿算子链反向传导，最终拖慢整个作业甚至 Source 停止拉数据。
- **典型根因**：某个算子计算重 / 访问外部系统慢（维表 join、写下游 DB）、数据倾斜导致单 SubTask 热点、资源不足（CPU / 网络）、GC 停顿。
- **关键指标 / 取证**：Flink Web UI 的 Backpressure 监控、`busyTimeMsPerSecond`、各算子 `numRecordsInPerSecond` 落差、SubTask 级别的延迟分布、TaskManager GC 时间。
- **排查思路**：从 Source 往下游找第一个「忙碌但输出跟不上」的算子，它就是瓶颈；再判断是倾斜（单并发热）还是整体慢（所有并发都慢）。
- **易追问**：反压会不会丢数据？——不会，Flink 靠信用值（credit-based）流控反压，会让 Source 降速，但 Checkpoint 持续超时才会真正出问题。

### Flink Checkpoint 失败 / 超时

- **一句话**：Flink 靠周期性 Checkpoint 做容错，Checkpoint 持续失败意味着作业一旦重启会丢大量状态或无法恢复。
- **典型根因**：反压导致 Barrier 对齐慢（Barrier 卡在某个慢算子）、状态太大、状态后端（如 RocksDB）写盘慢 / 远端存储（OSS/HDFS）抖动、对齐超时。
- **关键指标**：Checkpoint 耗时、对齐时间（alignment duration）、状态大小增长趋势、失败次数。
- **排查思路**：先看是「对齐慢」还是「持久化慢」——对齐慢往往是反压的连带症状，持久化慢看状态后端和远端存储；Unaligned Checkpoint 可缓解对齐慢但增加状态体积。
- **易追问**：Checkpoint 和 Savepoint 区别？——Checkpoint 是引擎自动容错、格式可能与版本耦合；Savepoint 是用户主动触发、用于升级 / 迁移。

### Flink 状态后端膨胀

- **一句话**：有状态作业（窗口、聚合、去重）的状态会随 key 基数无限增长，没有 TTL 或清理就会撑爆。
- **典型根因**：没设状态 TTL、key 基数爆炸、大窗口、RocksDB Compaction 跟不上。
- **排查思路**：看状态大小趋势、Checkpoint size、RocksDB 指标；治理靠 TTL、增量 Checkpoint、合理的 key 设计。

### Kafka Consumer Lag 积压

- **一句话**：消费位点（committed offset）和最新位点（log end offset）的差值持续拉大，说明消费跟不上。
- **典型根因（必须区分两类）**：
  - **消费能力不足**：消费逻辑慢（下游写库慢、单条处理重）、分区数 < 消费者并发上限、消费者 GC / Full GC。
  - **消费停了**：消费者频繁 Rebalance、卡在某条消息（poison message）、消费者实例挂了没拉起、位点提交失败导致重复 / 卡住。
- **关键指标**：每个 partition 的 lag、消费速率（records-consumed-rate）、`max.poll.interval` 超时、Rebalance 频率。
- **排查思路**：先看 lag 是「整体均匀增长」（能力不足）还是「个别 partition 不动」（消费者卡死 / 分区倾斜）；再看 Rebalance 日志和消费者存活。
- **易追问**：怎么扩消费能力？——加消费者实例直到等于分区数（再多无效，受分区数上限约束），或加分区数（但会打乱 key 顺序）。

### Kafka Rebalance 风暴

- **一句话**：Consumer Group 成员变动会触发分区重新分配，频繁 Rebalance 期间消费暂停，表现为周期性 lag 抖动。
- **典型根因**：消费者处理超过 `max.poll.interval.ms` 被踢出、心跳超时、消费者频繁重启 / 扩缩容、`session.timeout` 配置不合理。
- **排查思路**：看 Rebalance 触发原因日志、单次 poll 处理时长是否超阈值；治理靠减小单批处理量、调大 poll interval、用 Static Membership 或 Cooperative Rebalance 减少 Stop-the-world。

### Kafka ISR 收缩 / Under-replicated Partitions

- **一句话**：ISR（In-Sync Replicas）是与 Leader 保持同步的副本集合，ISR 收缩意味着副本同步跟不上，可用性和持久性下降。
- **典型根因**：Broker 慢盘 / 磁盘满、网络抖动、Follower 拉取跟不上、Broker GC、分区 Leader 倾斜。
- **关键指标**：`UnderReplicatedPartitions`、`OfflinePartitionsCount`、ISR 收缩频率、各 Broker 磁盘 / 网络。
- **易追问**：`acks=all` 和 ISR 关系？——`acks=all` + `min.insync.replicas` 决定了写入要等多少副本确认，ISR 不足时生产者会被拒绝写入（保一致性牺牲可用性）。

### Spark 数据倾斜

- **一句话**：某些 key 的数据量远超其他，导致个别 task 处理时间极长，拖垮整个 Stage。
- **典型根因**：join / groupBy 的 key 分布不均（热点 key、null key 集中）、分区数不合理。
- **关键指标**：Spark UI 里 Stage 的 task 时间分布（max 远大于 median）、shuffle read 数据量分布。
- **治理**：加盐打散热点 key、broadcast join 小表、AQE（Adaptive Query Execution）自动处理倾斜、调分区数。
- **和我经验映射**：我在 Soda Pepsi 做 Spark on K8s 落地，倾斜表现为某个 Executor Pod 长时间不退、整个任务卡在最后一个 Stage。

### Spark Shuffle / OOM / Executor 丢失

- **一句话**：Shuffle 是宽依赖的数据重分布，数据量大或内存不足时会溢写磁盘甚至 OOM 导致 Executor 被杀。
- **典型根因**：分区过大、shuffle 数据量大、Executor 内存 / overhead 配置不足、倾斜叠加。
- **排查思路**：看 Executor 日志的 OOM / `Container killed by YARN/K8s`、shuffle spill 量、GC 时间；在 K8s 场景还要看 Pod 是否因 memory limit 被 OOMKill。
- **和我经验映射**：Spark on K8s 下 Executor 是 Pod，OOM 表现为 Pod 被 OOMKill、driver 报 Executor lost，要结合 K8s 事件和 Spark UI 一起看——这是我真实排查过的场景。

### HDFS NameNode 压力 / 小文件问题

- **一句话**：NameNode 把所有文件 / Block 元数据放内存，小文件过多会撑爆内存、拉长 GC、拖慢所有元数据操作。
- **典型根因**：海量小文件（流式写、未合并的分区）、客户端高并发元数据请求、NameNode GC。
- **关键指标**：NameNode 堆内存 / GC、RPC 队列长度 / 处理延迟、Block 数量、文件数趋势。
- **治理**：小文件合并、归档（HAR）、控制分区粒度、用对象存储替代。
- **和我经验映射**：MRS / OBS 场景下，存储底座换成华为对象存储，小文件和元数据压力的表现方式会变，但「公共存储依赖放大故障」的逻辑一致。

### HDFS Block 丢失 / 副本不足 / 慢盘

- **一句话**：DataNode 故障或磁盘损坏会导致副本不足，慢盘会拖慢读写但不报错，是最难定位的一类。
- **排查思路**：`fsck` 看 Missing / Under-replicated Block、看 DataNode 存活和磁盘健康、慢盘靠磁盘 IO 延迟指标定位。

### 批处理调度积压（我的真实主场）

- **一句话**：调度系统里大量任务实例堆积、产出 SLA 破线。这是我在 Soda Pepsi 真实治理过的。
- **典型根因（必须分类）**：
  - **依赖未就绪**：上游任务没跑完 / 上游数据没到，下游卡在依赖判断。
  - **资源不足**：调度队列 / 资源池打满，任务排队拿不到 Worker / GPU / 计算资源。
  - **单点慢任务阻塞**：一个长尾任务占住资源 / 队列，后面排队。
  - **变更引入的批量失败**：一次任务变更或镜像变更导致一批任务失败重试，放大积压。
  - **Worker 执行异常**：Worker 容器启动失败、镜像拉取慢、Kerberos 认证失败、依赖缺失。
- **排查思路**：先定位积压是「卡在依赖」「卡在资源」还是「卡在执行」——这是分流的第一刀；再看是全局积压（资源 / 调度器问题）还是局部积压（单任务 / 单依赖链问题）。
- **和我经验映射**：我做过调度队列与资源选择、依赖判断、多引擎执行入口、失败处理协议，这块可以讲得很实。

### Hive Metastore 慢 / 元数据治理

- **一句话**：Metastore 是 Hive / Spark SQL 的元数据中心，它慢了表现为「所有 SQL 都卡在编译 / 解析期」，是隐藏单点。
- **典型根因**：底层 MySQL 慢查询 / 连接池打满、分区数爆炸（百万级分区）、高并发 DDL、Schema 变更锁。
- **排查思路**：看 Metastore RPC 延迟、底层 DB 慢查询、连接数、分区数；很容易被误判成「引擎慢」，要会区分「编译期慢」和「执行期慢」。

## 如果让我落地：把组件故障模式接入 AIOps 根因定位

这是把本篇和岗位（AIOps Agent）连起来的关键章节。前提是「假设落地」，不是「我已经建成」。

落地思路是：**每类组件的故障模式 = 一组可取证的指标 + 一棵决策树 + 一个 SOP**，Agent 负责编排取证而不是凭空推理。

1. **数据接入**：为每类组件接入标准化指标——Flink（反压 / Checkpoint / 状态大小）、Kafka（lag / Rebalance / ISR）、Spark（Stage task 分布 / shuffle / Executor 事件）、HDFS（NameNode GC / RPC / Block）、调度（积压量 / 依赖状态 / 资源水位）、Metastore（RPC 延迟 / DB 慢查询）。
2. **故障模式编码成知识**：把上面每个核心概念的「典型根因 + 关键指标 + 决策树」沉淀成 Agent 可检索的 SOP / Skill，让根因定位有据可依，而不是让大模型自由发挥。
3. **取证工具化**：Agent 调用工具拉指标、查日志、看变更记录、查依赖拓扑，形成证据链；比如 Flink 反压先自动拉算子级 busy 指标定位瓶颈算子。
4. **变更关联**：把组件故障和近期变更（任务上线、配置变更、版本升级、扩缩容）做时间和拓扑关联——这是大数据平台故障的高频根因，也对应 JD 的「变更风险」。
5. **分层定位**：先判断故障在「调度层 / 计算引擎层 / 存储层 / 元数据层」哪一层，再下钻，避免在错误的层里打转。
6. **受控输出**：Agent 给出根因假设 + 证据 + 建议动作（如重启、扩容、合并小文件），但执行动作受权限和审批约束，允许「证据不足，建议人工介入」。
7. **公共依赖优先**：HDFS / Metastore / 调度器这类公共依赖一旦异常会放大成大面积故障，Agent 应优先排查公共依赖，避免被海量上层告警淹没。

## 如果线上出问题，我怎么排查（通用路径）

按「分层 + 分流」的顺序，不在错误的层里耗时间：

1. **先定位现象层**：是产出延迟（批 / 流）、SQL 变慢、还是任务大面积失败？
2. **判断是公共依赖还是单作业**：看是否多个不相关作业同时异常——是则优先查 HDFS / Metastore / 调度器 / 网络等公共依赖。
3. **流处理**：Flink 看反压定位瓶颈算子 → 看是倾斜还是整体慢 → 看 Checkpoint 是否连带失败 → 看状态大小。
4. **消息**：Kafka 看 lag 是均匀增长还是个别 partition 不动 → 区分消费能力不足 / 消费停了 / Rebalance / ISR。
5. **批处理**：Spark 看 Stage task 时间分布定位倾斜 → 看 shuffle / OOM / Executor 事件 → K8s 场景叠看 Pod OOMKill 和事件。
6. **存储**：HDFS 看 NameNode GC / RPC 队列 / Block 健康 / 慢盘。
7. **调度**：积压先分「依赖 / 资源 / 执行」三类，再看全局还是局部。
8. **关联变更**：任何一层定位后都回看近期变更，变更是大数据平台最高频的根因之一。
9. **回到平台层**：把根因转成用户 / 值班能理解的结论和建议动作。

## 高频 QA

### 反压是什么，会不会丢数据

下游算子处理跟不上上游，压力沿算子链反向传导。Flink 靠 credit-based 流控反压，正常不丢数据，会让 Source 降速；真正的风险是反压连带 Checkpoint 持续超时，重启时恢复出问题。

### Kafka 消费积压的根因怎么分类

先分两大类：消费能力不足（lag 均匀增长，消费逻辑慢 / 并发不够 / GC）和消费停了（个别 partition 不动、频繁 Rebalance、消费者卡死或挂了）。这两类处置完全不同，分错方向就白排查。

### 怎么扩 Kafka 消费能力

加消费者实例直到等于分区数，再多无效（受分区数上限约束）；或者加分区数，但会打乱 key 顺序、影响有序消费。所以分区数要在设计期就留余量。

### Spark 数据倾斜怎么定位和治理

Spark UI 看 Stage 的 task 时间 / 数据量分布，max 远大于 median 就是倾斜。治理：热点 key 加盐打散、小表 broadcast join、开 AQE 自动处理倾斜、调分区数。

### Spark on K8s 下 Executor OOM 怎么排查

driver 报 Executor lost，要叠看两边：Spark UI 看 shuffle spill 和 GC，K8s 看 Pod 是不是因为 memory limit 被 OOMKill、看 Pod 事件。容器运行时下内存账本是 Pod limit，不是只看 Spark 配的 executor memory，还要算 overhead。这是我真实排查过的。

### HDFS 小文件为什么是大问题

NameNode 把所有文件 / Block 元数据放内存，海量小文件撑爆内存、拉长 GC、拖慢所有元数据操作，进而拖垮上面所有 Spark / Hive。治理靠合并、归档、控制分区粒度。

### 批处理调度积压怎么排查

先分三类：卡依赖（上游没就绪）、卡资源（队列 / 资源池打满）、卡执行（Worker 启动失败 / 慢任务阻塞）。再看是全局积压（调度器 / 资源问题）还是局部积压（单任务 / 单依赖链）。这是我在 Soda Pepsi 真实做过的。

### Hive Metastore 慢会有什么表现

表现为所有 SQL 都卡在编译 / 解析期，很容易误判成引擎慢。根因常是底层 MySQL 慢查询 / 连接打满、分区数爆炸、DDL 锁。要会区分「编译期慢」和「执行期慢」。

### 大数据平台故障最高频的根因是什么

变更和公共依赖。变更（任务 / 配置 / 版本 / 扩缩容）引入的批量失败，和 HDFS / Metastore / 调度器这类公共依赖抖动放大成大面积故障。所以排查时优先关联近期变更、优先查公共依赖。

### 这些组件故障怎么接入 AIOps Agent

把每类故障模式编码成「关键指标 + 根因决策树 + SOP」，让 Agent 调工具拉指标 / 查日志 / 关联变更 / 看依赖拓扑形成证据链，先分层（调度 / 计算 / 存储 / 元数据）再下钻，给根因假设和建议动作，执行受权限审批约束。核心是证据驱动，不让大模型自由发挥。

### 你没直接运维过 Flink / Kafka，为什么还懂这些

因为故障模式是工程通识，而且和我真实做的调度平台、Spark on K8s 是同一类问题——都是任务 / 资源 / 依赖 / 公共底座的稳定性。我从平台侧排查任务失败、和数据团队协作定位问题时，反复接触这些组件的故障表现。我能讲清楚故障模式和排查路径，但不会把自己包装成引擎集群的运维 owner。

### 你的经验和这个岗位的大数据平台匹配在哪

匹配在「平台控制面 + 调度治理 + 可观测 + AIOps Agent」这条线：我做过万级离线任务的调度平台、Spark on K8s 落地、跨云存储适配、生产变更管控，这些正是大数据平台稳定性的核心场景；再叠加我在 AIOps Agent 工程化上的准备，能把「懂故障模式」和「能用 Agent 自动定位」结合起来。
