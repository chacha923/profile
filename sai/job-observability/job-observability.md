# 面试定位卡

- **技术点**：TFJob 可观测
- **所属领域**：Kubernetes / AI Infra / 训练平台
- **面试价值**：体现从“设备监控”升级到“作业级诊断闭环”的平台能力
- **常见考法**：指标来源、误报控制、归因链路、成本与性能开销、自动化边界
- **适合挂钩项目**：SAI 训练平台、GPU 资源治理、分布式训练稳定性建设
- **不适合夸大的地方**：不夸大“全自动 RCA/自愈”，不夸大“控制面天然有训练语义指标”

# 三十秒回答

TFJob 可观测的核心不是多接几个 GPU 指标，而是把 TFJob 生命周期、Pod/Event、训练进程、数据 IO、GPU/通信统一到 Job 视角。  
当前阶段我优先依赖 TensorBoard 做低侵入核心指标观测，先回答“是否推进、是否变慢、是否可恢复”，再逐步增强归因和治理闭环。

# 为什么需要它

- **没有它之前的问题**：只能看到 GPU 利用率或日志片段，无法稳定回答“训练到底有没有推进、为什么慢、为什么失败”。
- **它的解决方式**：建立 Job 维度统一视图，把状态、指标、事件、日志关联成诊断闭环。
- **它引入的新问题**：指标高基数、采集开销、规则误报、跨数据源时间对齐复杂。
- **必须关注的场景**：Running 但无进展、Worker 反复重启、GPU 利用率低、Pending 卡住。

# 核心概念

- **作业视角可观测**
  - 解释：以 Job 为主键聚合状态、指标、事件、日志。
  - 面试展开点：为什么不能只看节点或设备。
- **训练语义指标**
  - 解释：`global_step`、`samples_per_sec`、`checkpoint_age` 等。
  - 面试展开点：控制面为什么没有这些指标。
- **设备指标**
  - 解释：DCGM 输出 GPU 利用率、显存、XID/ECC 等。
  - 面试展开点：能看到现象，但不能直接完成归因。
- **状态机观测**
  - 解释：TFJob condition、角色副本状态、phase 时长。
  - 面试展开点：生命周期异常如何快速定位。
- **组合判定**
  - 解释：多指标联合判断异常，而非单阈值。
  - 面试展开点：如何降低误报漏报。
- **分层存储**
  - 解释：指标库、事件库、日志归档分层。
  - 面试展开点：成本治理与高基数控制。

# 原理模型

![TFJob 可观测分层图](https://cdn.nlark.com/yuque/0/2026/png/29786964/1779257026861-d092d179-c8f5-4f94-879e-b040cb9adb4e.png)

## 底层 / 基础设施层

- Kubernetes 提供 TFJob CR、Pod、Event、Node 状态。
- DCGM 提供 GPU 设备层指标。
- NAS/OSS 提供训练日志、TensorBoard event、checkpoint 存储。

## 容器 / Kubernetes 层

- TFJob Operator 负责副本创建、状态推进、失败重试策略执行。
- Pod 生命周期和 Event 提供调度与运行异常证据。

## AI Infra 层

- 平台 watcher 采集 TFJob/Pod/Event，并统一 `job_id` 维度标签。
- TensorBoard event 解析链路提供低侵入训练进程指标输入。
- 规则引擎把状态与指标组合成可解释告警。

## 应用 / 业务层

- 算法侧通过 TensorBoard 看训练细节。
- 平台侧通过 Job 视图做稳定性、吞吐和故障治理。

# 关键机制

![TFJob 可观测演进架构](https://cdn.nlark.com/yuque/0/2026/png/29786964/1779257036953-1a94cbe1-55be-4db9-bc7f-0d834cfc191b.png)

## 机制一：Job 维度标签归一

- **解决的问题**：跨 TFJob、Pod、GPU、日志的数据无法直接关联，排障链路断裂。
- **工作方式**：统一 `cluster/namespace/job/role/replica` 作为主关联标签；高基数标签（如 `pod/gpu_uuid`）下沉到事件和日志明细层。
- **代价**：需要治理标签口径和存储分层，否则 Prometheus 基数容易失控。
- **面试追问**：你怎么避免全量标签进入指标系统导致成本爆炸？

## 机制二：TensorBoard 核心指标分级采集

- **解决的问题**：控制面没有训练语义指标，无法判定“Running 但无进展”。
- **工作方式**：优先解析 P0 指标：`global_step`、`step_time`、`samples_per_sec`、`loss`、`checkpoint_age`；P1/P2 指标按需开启，不做全量常驻。
- **代价**：event 解析通常是分钟级近实时，且依赖业务写入规范，不如统一 SDK 稳定。
- **面试追问**：如果业务没写 TensorBoard scalar，你的兜底方案是什么？

## 机制三：组合规则判定与阶段豁免

- **解决的问题**：单指标告警误报高，容易把 eval/checkpoint/warmup 误判成故障。
- **工作方式**：用 `step 增量 + 吞吐更新 + checkpoint 心跳` 组合判定；对 eval、长 checkpoint、warmup 设置豁免窗口；告警分提示级和故障级。
- **代价**：规则参数需要持续回放校准，否则容易出现跨任务类型泛化不足。
- **面试追问**：你如何量化误报率，并持续优化规则阈值？

## 机制四：症状到动作的闭环治理

- **解决的问题**：监控系统只报现象，不给可执行动作，人工排障慢。
- **工作方式**：按“症状 -> 假设 -> 验证 -> 归因 -> 动作 -> 复测”输出诊断建议；仅对可恢复故障做自动化动作。
- **代价**：需要维护归因知识库和动作安全闸，建设成本高于纯监控。
- **面试追问**：哪些场景可以自动重试，哪些必须人工确认？

# 横向对比

- **DCGM vs TensorBoard**
  - 区别：DCGM 是设备视角，TensorBoard 是训练语义视角。
  - 使用时机：设备健康看 DCGM，训练推进看 TensorBoard。
  - 注意点：不要把通信语义误说成 DCGM 原生能力。
- **TensorBoard 解析 vs 业务 SDK 埋点**
  - 区别：解析低侵入、标准化弱；SDK 标准化强、接入成本高。
  - 使用时机：存量优先解析，新任务逐步 SDK 化。
  - 注意点：先低侵入落地，再逐步标准化。
- **单阈值告警 vs 组合规则**
  - 区别：单阈值简单但误报高；组合规则准确但复杂。
  - 使用时机：核心故障建议组合规则。
  - 注意点：必须讲豁免窗口和告警分级。
- **全量采集 vs 分级采集**
  - 区别：全量信息丰富但成本高；分级更稳健。
  - 使用时机：平台初期用 P0/P1 分级。
  - 注意点：要说明高基数治理策略。

# 典型业务场景

- **Running 但无进展**
  - 相关性：最常见“假健康”场景。
  - 现象：Pod Running，但模型不收敛或 step 不动。
  - 排查：看 step 增量、吞吐、checkpoint 心跳。
  - 优化：调整数据链路、训练参数、故障快速失败。
- **Worker 频繁重启**
  - 相关性：影响训练稳定性与 GPU 占用。
  - 现象：重启次数高、退出码异常。
  - 排查：结合 Pod lastState、Event、关键日志。
  - 优化：分类治理资源、业务、节点、存储问题。
- **GPU 利用率低**
  - 相关性：容易误判为设备问题。
  - 现象：util 低但任务未失败。
  - 排查：联合看 data_wait、CPU、step_time。
  - 优化：优化输入 pipeline、预处理、并行度。
- **Pending 卡住**
  - 相关性：直接影响任务时效。
  - 现象：长时间 Pending。
  - 排查：看 Event 的调度/镜像/PVC 原因。
  - 优化：队列治理、资源规格调整、节点池策略。

# 排障路径

![TFJob 诊断闭环流程图](https://cdn.nlark.com/yuque/0/2026/png/29786964/1779259142517-624d3282-5481-4ac2-b632-f71cee35cbaa.png)

- **症状**：TFJob Running 但吞吐明显下降。
- **初始假设**：输入慢、预处理慢、通信等待、checkpoint 慢、设备异常。
- **验证命令**：
  - `kubectl get tfjob -n <ns> <job> -o yaml`
  - `kubectl get pod -n <ns> -l training.kubeflow.org/job-name=<job>`
  - `kubectl describe pod <pod> -n <ns>`
- **关键指标**：`global_step`、`step_time`、`samples_per_sec`、`checkpoint_age`、GPU util、Pod 重启数。
- **可能结论**：IO 瓶颈、CPU 预处理瓶颈、通信等待、存储写入慢、设备异常。
- **优化动作**：调整数据加载并发、缓存策略、batch 配置、checkpoint 策略、故障重试策略。
- **复测方式**：对比优化前后 30 分钟窗口的 step 增量和吞吐恢复情况。

```bash
kubectl get tfjob -n <ns> <job> -o yaml
```

这条命令用于验证什么：验证 TFJob condition、角色副本状态、重试策略状态。  
重点看什么：`status.conditions`、`replicaStatuses`、`startTime/completionTime`。  
异常说明什么：状态卡住或副本异常，问题在控制面或编排层。

```bash
kubectl describe pod <pod> -n <ns>
```

这条命令用于验证什么：验证容器退出原因、调度事件、镜像和存储异常。  
重点看什么：`Last State`、`Exit Code`、`Events`。  
异常说明什么：可快速区分资源、节点、镜像、PVC、业务崩溃等路径。

# 风险、边界和误区

- **误区一**：“控制面就能看训练是否推进”
  - 问题：控制面只有对象状态，没有训练语义。
  - 稳妥表达：训练推进要靠 TensorBoard/SDK 等进程侧指标补齐。
- **误区二**：“GPU util 低就是 GPU 故障”
  - 问题：容易忽略 IO/CPU/通信瓶颈。
  - 稳妥表达：先做组合归因，再判断设备侧问题。
- **误区三**：“把所有日志和 trace 都采上来”
  - 问题：成本和性能风险高。
  - 稳妥表达：先抓 P0 核心指标，按诊断窗口扩展 P1/P2。
- **误区四**：“告警触发就自动重试”
  - 问题：可能放大故障，浪费 GPU。
  - 稳妥表达：仅可恢复故障自动化，其他场景给建议并人工确认。
- **误区五**：“TensorBoard 可以替代一切可观测”
  - 问题：TB 更偏训练分析，不是完整治理系统。
  - 稳妥表达：TB 是训练语义核心输入之一，需与状态和事件结合。

# 高频 Q&A

## 为什么不只依赖 GPU 指标？

因为 GPU 指标只反映设备状态，不能直接反映训练语义。训练是否推进必须结合 step、吞吐和 checkpoint 心跳判断。

## 控制面为什么拿不到训练进程指标？

控制面主要管理资源对象生命周期，不感知训练循环语义，所以不会天然产出 step 级指标。

## 训练进程指标必须业务重埋点吗？

不一定。当前阶段可优先解析 TensorBoard event 低侵入落地，再对新任务逐步推进统一 SDK 契约。

## TensorBoard 优先看哪些核心指标？

P0 优先 `global_step`、`step_time`、`samples_per_sec`、`loss`、`checkpoint_age`，先回答推进、吞吐和恢复风险。

## 如何避免“Running 但无进展”误报？

使用组合规则，不用单指标阈值，并给 eval、long checkpoint、warmup 设置豁免窗口和告警分级。

## 为什么不全量开 profiler？

profiler 开销高，常驻会影响训练性能和成本，适合故障窗口定向开启，不适合作为基础采集。

## 高基数标签怎么治理？

低基数标签进指标系统做告警，高基数明细下沉事件和日志库，并配合 TTL、聚合视图和降采样。

## 如何证明体系有效？

看量化结果：MTTR、误报率、归因覆盖率、GPU 空转时长、失败任务提前终止率。

# 三档背诵版

## 三十秒版

TFJob 可观测要从设备监控升级到作业级闭环。当前我优先基于 TensorBoard 做低侵入核心指标观测，结合 TFJob/Pod/Event 建立组合规则，先解决“是否推进、是否变慢、是否可恢复”。

## 三分钟版

我们原有 TensorBoard、NAS 日志和 DCGM，但缺少 Job 统一视角，导致很多问题只能看现象。  
我把可观测拆为状态、训练语义、IO、设备四条主线，统一到 Job 维度。  
训练语义当前优先用 TensorBoard 核心指标：`global_step`、`step_time`、`samples_per_sec`、`loss`、`checkpoint_age`。  
告警采用组合规则并设置阶段豁免，避免把 eval 或 checkpoint 误判成故障。  
最终目标是形成“症状 -> 归因 -> 动作 -> 复测”的治理闭环。

## 五分钟版

问题本质是：没有训练语义就无法完成平台级归因。  
控制面只知道对象状态，DCGM 只知道设备状态，二者都不能单独回答训练是否推进。  
因此我先做低侵入路径：统一标签、打通 TFJob/Pod/Event/DCGM，再把 TensorBoard 核心 scalar 转成平台指标。  
在规则层用 `step 增量 + 吞吐 + checkpoint 心跳` 组合判定异常，并对 eval、warmup、长 checkpoint 做豁免。  
在成本层实施标签分层和分级采集，避免高基数与性能开销失控。  
自动化只覆盖可恢复故障，其他场景给可解释结论和治理建议。  
评估效果用 MTTR、误报率、归因覆盖率和 GPU 空转时长，证明平台治理能力提升而不只是“多了几个图表”。
