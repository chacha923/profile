# 项目定位

- 项目名称：AIOps 智能运维告警根因分析系统（内部代号「拉普拉斯」）
- 代码位置：`/opt/coding/code.soulapp-inc.cn/devops/aiops`
- 所属领域：AIOps、稳定性平台、LLM Agent 工程、可观测、K8s 排障
- 项目类型：面向 K8s/容器告警的自动取证与根因分析 Agent，挂在告警链路下游
- 核心目标：把告警从「通知到人」推进到「先自动诊断再到人」，由 LLM Agent 调用只读运维工具取证，对照知识库故障案例做根因匹配，输出结构化分析报告回贴飞书。
- 服务对象：SRE、值班人员、业务研发、平台运维
- 关键词：ReAct、Think-Act-Observe、多 Agent 圆桌、Function Calling、只读取证工具、知识库故障案例、防幻觉质检、AI Gateway、RocketMQ、飞书回写
- 面试价值：能把传统告警平台、K8s 排障和大模型 Agent 串起来讲清楚，体现「智能运维不是聊天机器人，而是证据驱动、受控、可质检的排障系统」的工程理解，同时覆盖 Agent 编排、工具设计、上下文与防幻觉这些当下最被追问的点。

一句话讲法：这套系统是公司告警链路下游的根因分析 Agent，收到容器类告警后，由 LLM Agent 按 ReAct 范式边推理边调只读工具（describe_pod、Prometheus 指标、历史事件、Pod 日志、变更记录、节点池状态等）取证，对照飞书知识库里的「故障案例分析」逐项验证现象，匹配出唯一根因和解决方案，再经过一套 12 项否决式质检后输出结构化报告，回贴到飞书告警消息。我参与的是诊断 Agent 核心链路的建设与改造，包括 ReAct 引擎、工具集、知识约束和多 Agent 编排。

核心技术锚点：

- 两种 Agent 形态：生产主力是单 Agent ReAct（`single_agent.py` 的「拉普拉斯」），另有一套多 Agent 圆桌协作（`multi_agent.py` 的 receiver/analyst/planner/k8s_engineer/examiner 五角色）。
- ReAct 引擎：`TAOPlayer` 实现 Think-Act-Observe 循环，带轮数上限、工具错误重试、状态机和强制汇总，所有工具走 OpenAI Function Calling 协议。
- 证据驱动、防幻觉：`temperature=0`，工具全部只读取证，输出强制走「故障根因/分析凭据/解决方案/参考链接」模版，匹配不上知识库案例就输出「未找到根因」，严禁推断。
- 知识约束：诊断逻辑不靠模型自由发挥，而是对照飞书知识库「故障案例分析」表逐项验证「故障现象」，形成「时间→现象→证据」链条。
- 质检闭环：`OBSERVER_PROMPT` 是一套 12 项逐项否决式检查清单，任意一项不通过禁止调用 `terminate` 结束。
- LLM 解耦：从 OpenAI SDK 直连多模型（gpt-4o/doubao/deepseek-r1）演进到统一走内部 AI Gateway（doubao-seed-2.0），自封装 HTTP 调用并转回 OpenAI 兼容结构。

# 项目背景

- 容器告警止于通知：OWL/告警网关把「容器异常重启」「容器 OOMKilled」「容器创建 Pending」这类告警推给值班人后，后续的查 Pod、看事件、拉指标、翻日志、对历史案例都靠人脑和手工，重复且慢。要解决的方向是把这套固定的取证动作沉淀成工具，让 Agent 先跑一遍。
- 排障经验散落：同一类告警的排查路径、判定标准、解决方案散落在个人经验和聊天记录里，没有结构化沉淀。要解决的方向是把故障案例做成知识库表（故障现象、直接根因、解决方案、特别说明），让诊断有据可依。
- LLM 直接问诊不可靠：如果只把告警丢给大模型让它「分析一下」，会编造数据、给「可能需要检查 xxx」这类不可执行的废话，甚至虚构操作路径。要解决的方向是用工程手段约束模型：只读取证、强制数据溯源、逐项质检、单一根因。
- 诊断要可控可审计：运维场景下，Agent 不能放任执行写操作，结论也要能追溯到工具原始输出。要解决的方向是工具全部只读、报告里禁止出现工具名和命令行、所有结论必须挂在具体指标/日志/事件上。

这个项目不是「接个大模型做告警问答」，而是把根因分析拆成几层可控环节：接入层从告警网关/MQ 取告警并解析关键字段，取证层用只读工具采集 Pod/指标/事件/日志/变更证据，知识层对照故障案例做现象逐项验证，推理层用 ReAct 边想边查，质检层用否决清单兜底，输出层产出结构化报告并回写飞书。

# 项目目标

- 自动取证：把容器告警的固定排查动作（看 Pod、查事件、拉指标、翻日志、看变更、看节点池）沉淀成只读工具，由 Agent 自动调用。
- 知识驱动：诊断对照飞书知识库「故障案例分析」逐项验证现象，匹配出直接根因，匹配不上如实输出「未找到根因」。
- 证据闭环：每条结论都要挂在具体指标数值、日志片段、事件记录上，禁止主观推断和条件式结论。
- 输出规范：强制结构化报告模版（故障根因/分析凭据/解决方案/参考链接），面向研发可读、可执行，不暴露 kubectl 命令和工具名。
- 受控质检：终止前过一套 12 项否决式检查，任一不通过不允许结束，降低幻觉和半成品报告。
- 形态可选：既支持单 Agent ReAct 端到端跑通，也支持多 Agent 圆桌把「提因→规划→执行→裁决」拆给不同角色。
- 链路解耦：诊断挂在告警链路下游异步触发，失败只记日志/告警，不影响主告警通知。

## 架构分层

- 接入触发层：`views.EventAnalysisView`（HTTP POST 同步触发）、`entrys.start`（轮询 alarm-gateway 拉近 1 分钟告警）、`brain` 的 RocketMQ `PushConsumer('AI_ALERT_GROUP')`（消息触发）。负责拿到告警、解析告警类型/时间/服务树/Pod 名等关键字段。
- 推理内核层：
  - 单 Agent：`single_agent.start_analysis` + `TAOPlayer`（角色「拉普拉斯」），生产主力链路。
  - 多 Agent：`multi_agent.AsyncRoundTable` + 五个 `TAOPlayer` 角色，协作式根因分析。
  - assistant 模式：`brain.inquiry` + `AI`/`get_assistant_answer`，多专业助手（philosopher/k8s/prom/rocketmq）协作的早期形态。
- LLM 接入层：`llms.py` 定义 openai/doubao/deepseek 三套 client；`TAOPlayer.create_completion` 当前走内部 AI Gateway（`prod-ai-gateway.soulapp-inc.cn`，doubao-seed-2.0），`create_completion_openai` 作为备用 OpenAI SDK 通道。
- 工具取证层（全部只读，langchain `StructuredTool`）：
  - k8s：`describe_pod`、`get_pod_logs`、`get_service_info`、`get_ingress_info`、`get_hpa_info`、`get_history_events`
  - prometheus：Pod/Node/NodePool/JVM/Service/通用 六类 `get_*_metric_data`
  - info：`get_pod_all_info`、`get_service_all_info`（聚合上下文）
  - log：`get_pod_history_log`（ClickHouse signoz/logging 历史日志）
  - changes：`get_service_change_history`（发布变更记录）
  - nodepool：`get_nodepool_status`
  - feishu：`read_feishu_document`（读知识库故障案例/SOP）
  - memory：`get_project_context`、`read_file_content`
  - sniffer：`tcp_sniffer`、`http_sniffer`；docusearch：Chroma 向量检索指标名；terminate：受控结束
- 知识约束层：飞书知识库「故障案例分析」表（故障类型/故障现象/直接根因/解决方案/特别说明）、节点池资源配置要求文档，是诊断的最高优先级依据。
- 输出与留痕层：结构化报告（故障根因/分析凭据/解决方案/参考链接）→ 过滤 markdown 包裹标记 → `send_clerk` 回贴飞书 → 写入 MySQL `aiops.analysis_history`（kind/target/result）。

## 模块边界

- `container/players.py`：Agent 内核，`TAOPlayer`（Think-Act-Observe/ReAct）与 `SinglePlayer`（单次思考），含状态机、轮数控制、工具执行、强制汇总。
- `container/single_agent.py`：单 Agent 生产链路，`start_analysis` 解析告警、拉取上下文、读知识库案例、构造 prompt、驱动 `TAOPlayer.think()`、落库。
- `container/multi_agent.py`：多 Agent 圆桌 `AsyncRoundTable`，五角色 Queue 协作、消息路由、终止判定、汇总。
- `container/brain.py`：assistant 协作形态入口，RocketMQ 消费 + 多专业助手。
- `container/prompts.py`：所有角色与流程 prompt（单 Agent 系统 prompt、五角色 prompt、`OBSERVER_PROMPT` 质检清单、`DEFAULT_NEXT_STEP_PROMPT`、`TOOLCALL_ERROR_PROMPT`）。
- `container/llms.py`：多模型 client 定义。
- `container/tools/*`：12 类只读取证工具 + terminate；`tools/base.py` 定义 `BaseTool`/`ToolResult` 抽象。
- `container/views.py`、`container/entrys.py`：HTTP 与轮询触发入口。
- `utils/*`：`hrt`（HTTP 封装）、`prom`、`kube_sdk`、`notifier`（send_clerk 飞书）、`db`（MySQL）、`ck`（ClickHouse）等基础设施。

# 核心能力

- 告警解析与上下文采集：从告警内容正则提取服务树（cluster.service）、Pod 名、告警时间，按告警类型预拉 `get_pod_all_info` / `get_service_all_info` 作为初始上下文。展开点是为什么要先固定提取关键字段，而不是把原始告警整个丢给模型。
- ReAct 自主取证：Agent 边推理边决定调哪个工具，基于上一步工具输出决定下一步，直到证据足够。展开点是 think→action→observe 循环、何时停止、怎么避免无效多轮。
- 故障案例匹配：对照知识库「故障案例分析」逐项验证「故障现象」，全部通过才认定该案例的「直接根因」。展开点是「时间→现象→证据」链条和单一根因原则。
- 结构化报告输出：强制模版（故障根因/分析凭据/解决方案/参考链接），面向研发可读可执行。展开点是为什么报告里禁止出现工具名和 kubectl 命令。
- 否决式质检：终止前过 12 项检查（时间戳格式、不确定表述、条件逻辑、数据支撑、逻辑一致、单一根因、数据溯源、证据完整、工具名替换、格式、完整性），任一不过禁止结束。展开点是质检怎么挂在 `terminate` 工具前。
- 多 Agent 协作：把「提出可能根因→规划验证步骤→执行验证→裁决汇总」拆给 analyst/planner/k8s_engineer/examiner。展开点是多 Agent 与单 Agent 的取舍。
- 知识库联动：工具输出里如果出现飞书知识库链接，Agent 会自动抓取并查阅文档内容再继续推理。展开点是这相当于一个轻量的 RAG/工具触发检索。
- 失败与边界处理：不支持的服务类型（nuclio、qa-）直接拒绝；匹配不到根因输出「未找到根因」；诊断异常 `send_clerk` 告警但不阻塞主链路。

# 核心流程

![核心流程](https://cdn.nlark.com/yuque/0/2026/png/29786964/1781436816609-078ccc5e-712c-4955-a7ca-894819dc308e.png)

## 单 Agent ReAct 主链路（生产主力）

- 触发：`EventAnalysisView.post` 或 `entrys.start` 拿到告警，组装成 `{alert_type, alert_time, owner, content}`。失败处理：不支持的 nuclio/qa 服务直接返回 400。
- 解析：`start_analysis` 用正则从 `content` 提取服务树 `cluster.service`，对容器类告警再提取 Pod 名；解析失败抛异常并带原文便于排查。
- 上下文预取：按告警类型调 `get_pod_all_info` / `get_service_all_info` 拿 Pod/服务聚合信息；`end_time` 取告警时间 +5 分钟，限定指标查询窗口。
- 知识库取案例：`get_alert_analysis(alert_type)` 读飞书知识库「故障案例」sheet，按告警类型筛出对应案例（故障现象/直接根因/解决方案/特别说明）。
- 构造 prompt：把告警信息、故障案例、初始 all_info、end_time 填进 `SINGLE_USER_PROMPT`，配 `SINGLE_SYSTEM_PROMPT`（拉普拉斯，强制分析流程与输出模版）。
- ReAct 循环：`TAOPlayer.think()` 启动，按需调只读工具取证（describe_pod、指标、事件、日志、变更、节点池），逐项验证故障现象。
- 质检收口：满足条件后注入 `OBSERVER_PROMPT` 做 12 项否决检查，全部通过才调 `terminate` 结束。
- 输出落地：去掉 markdown 包裹标记，写入 `aiops.analysis_history`，`send_clerk` 回贴飞书；含「未找到根因」或缺「分析报告」结构则按失败处理并告警。

## 多 Agent 圆桌链路

- 初始化：`init_round_table` 建 `AsyncRoundTable`，join 五个角色 Player，各自一个 `asyncio.Queue` 和 listen 协程，receiver 先把告警抛给 analyst。
- analyst 提因：按可能性从高到低输出最多 5 条根因（纯 JSON 数组），逐条转发给 planner。失败处理：JSON 解析失败直接报错。
- planner 规划：对每个根因输出验证步骤（JSON，含 title/description/is_parallel/executor），把每步派给指定 executor（如 k8s_engineer）。
- k8s_engineer 执行：用 k8s 工具按步骤验证，只依据工具输出给阶段性结论，把结论转给 examiner。
- examiner 裁决：判断结论是否为根因，是则「完成」，否则「收到」；最终读 summary 文件汇总最可能的 1 个根因+解决方案。
- 终止：当 k8s_engineer 队列空且根因/步骤验证都完成，或已产出最终答案，`listen_summary` 取消所有任务并让 examiner 汇总输出。

# 关键设计

## Think-Act-Observe（ReAct）引擎

![ReAct 引擎与状态机](https://cdn.nlark.com/yuque/0/2026/png/29786964/1781436828649-dfcbdaa8-0fdc-414f-a3d0-682c93eb3165.png)

解决的问题：容器告警的排查步数不固定，有的看一眼 Pod 事件就够，有的要翻指标再翻日志再看变更。固定流程不够灵活，纯放任模型又会绕路、漏证据、编数据。

方案设计：`TAOPlayer` 实现 think→action→observe 循环。

- think：调 LLM（带 tools）。若返回 tool_calls 就进 action；若只有文本结论就进 observe；轮数超过 `max_dialog_rounds=10` 则注入「思考超限，请汇总并按模版输出」强制收口。
- action：逐个执行工具调用，把 `执行命令X的观察输出:...` 回灌成 tool 消息；若工具结果里出现飞书知识库链接，自动追加「请查阅以下链接」再回到 think（轻量工具触发检索）。
- observe：根据状态决定下一步——上一条是 tool 结果就继续 think；检测到工具调用失败且未超 `max_tool_call_times=1` 就注入 `TOOLCALL_ERROR_PROMPT` 重试；否则注入 `OBSERVER_PROMPT` 质检后再 think。

为什么这样做：ReAct 让推理和取证交替进行，每一步行动都基于上一步证据，适合步数不定的排障；轮数上限和强制汇总保证一定会收敛出结果，不会无限循环。

取舍：多轮调用会增加 Token 和延迟；`max_tool_call_times=1` 容错偏弱，工具瞬时失败容易直接放弃。改造方向是把重试做成带退避的有限重试，并区分「工具不可用」和「参数错误」。

面试追问：怎么防止 Agent 绕路或不收敛？答轮数上限 + 强制汇总 + observe 阶段的质检注入；状态机 `IDLE/RUNNING/FAILED/FINISHED/ERROR` 控制生命周期；`terminate` 是唯一合法结束出口。

## 工具即取证：只读 + 结构化 + Function Calling

解决的问题：让模型「自己分析」会编数据，必须把真实的运维数据喂给它，并约束它只能拿数据、不能动线上。

方案设计：所有工具用 langchain `StructuredTool` 定义，`_format_tools_for_openai` 转成 OpenAI Function Calling schema 交给模型选择。工具覆盖 k8s 资源/事件/日志、Prometheus 多维指标、Pod/服务聚合信息、历史日志（ClickHouse）、变更记录、节点池状态、飞书知识库、记忆。全部只读取证，没有删除/重启/改配置类写操作。工具结果统一包装成 `执行命令X的观察输出:{result}` 回灌，空结果也作为合法返回（`命令X完成，没有输出`）。

为什么这样做：工具设计决定 Agent 上限。只读保证安全；结构化和「空结果合法」让模型能基于确定的证据推理，而不是因为拿不到数据就编。

取舍：只读意味着不能自动处置，只能给人建议；工具数量多会撑大 tools schema 和上下文。改造方向是按告警类型动态裁剪可用工具集、给工具输出加结构化 schema（而非纯字符串）。

面试追问：工具结果怎么避免污染上下文？答 `repr(result)[:N]` 截断日志、`_clean_messages` 规整消息结构、按需截断长输出；改造方向是引入证据对象和上下文裁剪。

## 知识约束 + 逐项现象验证（防幻觉核心）

![防幻觉四层](https://cdn.nlark.com/yuque/0/2026/png/29786964/1781436871873-59d374f3-1386-43f8-baad-560e90de3f9d.png)

解决的问题：大模型在没有约束时会「自由发挥」，给出看似合理但无依据的根因。

方案设计：诊断不靠模型记忆，而是对照飞书知识库「故障案例分析」表。`SINGLE_SYSTEM_PROMPT` 强制：自上而下对每个案例的「故障现象」逐项验证，每项必须记录「时间→现象→证据（指标/日志/事件）」；所有现象都通过才认定该案例的「直接根因」；任一现象验证失败或工具无数据就排除该案例继续下一个；所有案例都不通过则根因固定为「未找到根因」，严禁推断。还区分「直接根因」和基于日志深挖的「深层根因」，深层根因必须从直接根因延伸、不得归因到案例未提及的因素。

为什么这样做：把诊断从「开放式生成」收成「对照已知案例的验证」，每条结论都有出处，幻觉空间被压到最小。

取舍：只能诊断知识库已沉淀的故障类型，新型故障会落到「未找到根因」；知识库质量直接决定诊断能力。改造方向是把案例从飞书文档迁到可检索知识库 + 用 RAG 做相似案例召回。

面试追问：为什么宁可「未找到根因」也不让它猜？因为运维场景里一个错误根因比没有根因更危险，会误导处置；保守输出 + 人工兜底是更安全的设计。

## 12 项否决式质检（terminate 前置门）

解决的问题：模型生成的报告常有不确定表述（「可能」「建议检查」）、条件式结论（「若 xxx 则 xxx」）、数据对不上、暴露工具名/命令、格式不规范、被截断。

方案设计：`OBSERVER_PROMPT` 是一套 12 项逐项否决清单——报告可读性、分析彻底性、解决方案确定性、数据支撑、逻辑一致、单一根因、特别说明遵循、数据溯源、证据完整、内容有效性（工具名换成功能描述、删 kubectl 命令、删虚构路径）、报告格式、报告完整性。任意一项不通过禁止调用 `terminate`，必须修正后重出。`terminate` 工具参数要回传 `unverified_items`，未通过项非空则提示「请修正后输出最终报告」。

为什么这样做：把「质量门」做成模型必须自检的硬清单，而不是寄希望于一次生成就完美。

取舍：质检会再触发一轮模型调用、增加成本；清单偏长可能被模型部分忽略。改造方向是把部分检查项（时间戳格式、是否含命令行、是否含工具名）下沉成代码后置校验，而不是全交给模型自检。

面试追问：质检也是模型做的，怎么保证可信？答这是「软约束 + 代码兜底」的组合，纯规则能查的（命令行、工具名、markdown 包裹）应该用正则后置校验（代码里已有 `re.sub` 去 markdown 包裹的雏形），语义类的才交给模型自检。

## 多 Agent 圆桌的角色分工与终止判定

解决的问题：单 Agent 把提因、规划、执行、裁决都揽在一个上下文里，职责混、上下文长。多 Agent 想把它拆成专业角色协作。

方案设计：`AsyncRoundTable` 让五个 `TAOPlayer` 角色各持一个 `asyncio.Queue`，用 listen 协程收发消息：analyst 提因（最多 5 条 JSON）→ planner 对每条根因规划验证步骤（JSON，指定 executor）→ k8s_engineer 执行验证 → examiner 裁决并汇总。终止靠 `reason_verify_completed`、`step_verify_completed` 和 `k8s_engineer` 队列是否空，或 `has_final_answer`。

为什么这样做：角色分工让每个 Agent 的 prompt 更聚焦、上下文更短，提因和裁决分离也降低了「自己提因自己裁决」的偏置。

取舍：多 Agent 的并发用「线程 + asyncio.Queue + 轮询 sleep」实现，终止判定依赖多个布尔标志和队列空检查，时序较脆、不易调试；端到端延迟和 Token 成本都高于单 Agent。这也是我会重点讲的改造点：用更明确的编排框架（状态图/消息总线）替代手搓线程轮询。

面试追问：为什么生产主力是单 Agent 而不是多 Agent？答容器告警的诊断路径相对固定，单 Agent ReAct 足够且更稳、成本更低；多 Agent 更适合根因空间发散、需要多专业并行验证的复杂场景，当前更像探索形态。

# 技术难点

- 让模型证据驱动而非编造：难在大模型天然倾向「补全」出合理答案。解决思路是只读工具取证 + 强制数据溯源 + 逐项现象验证 + 否决式质检 + 「未找到根因」兜底。展开点是怎么一层层把幻觉空间压小。
- 步数不定的推理收敛：难在排查步数不固定，又不能让 Agent 无限循环。解决思路是 ReAct 循环 + 轮数上限 + 超限强制汇总 + 状态机 + 唯一 terminate 出口。展开点是 think/action/observe 的流转和停止条件。
- 工具结果与上下文管理：难在日志、事件、指标输出很长，多轮会撑爆上下文。解决思路是 `repr()[:N]` 截断、`_clean_messages` 规整、按需保留。展开点是当前是粗暴截断，改造方向是证据对象 + 上下文裁剪。
- 告警字段解析的脆弱性：难在告警内容是半结构化文本，靠正则提取服务树、Pod 名，格式一变就崩。解决思路是按告警类型分支提取 + 解析失败显式抛错带原文。展开点是为什么要让上游告警结构化，而不是下游硬解析。
- 多 Agent 并发与终止：难在五角色异步协作的时序和终止判定。解决思路是 Queue 隔离 + 布尔标志 + 队列空检查。展开点是当前实现的脆弱点和重构方向。
- LLM 供给的稳定性：难在多模型、多通道，且 Gateway 响应结构和 OpenAI SDK 不完全一致。解决思路是 `create_completion` 走 Gateway 后用 `SimpleNamespace` 把响应转回可点号访问的 OpenAI 兼容对象，`create_completion_openai` 留作备用。展开点是为什么要做这层适配。

# 稳定性与治理

- 推理收敛兜底：`max_dialog_rounds=10` + 超限强制汇总 + `get_summary` 兜底取含「故障根因/解决方案/分析凭据」的结论，保证一定有输出。
- 工具失败处理：`execute_tool` 捕获 JSON 解析错误和执行异常，回灌成 `工具调用失败:...`；observe 阶段检测失败并在 `max_tool_call_times=1` 内重试。
- 只读取证：全部工具无写操作，Agent 不能动线上资源，从源头规避误操作风险。
- 输出后置清洗：`re.sub` 去掉 `markdown/json/yaml` 等代码块包裹标记，避免污染回贴内容。
- 链路隔离：诊断挂告警下游异步触发，失败 `send_clerk` 告警但不阻塞主告警通知；不支持的 nuclio/qa 服务前置拦截。
- 结果留痕：`aiops.analysis_history`（kind=告警类型、target=告警内容、result=分析报告）落库，便于复盘和回溯。
- 确定性推理：`temperature=0`、`top_p=0.9`，降低同输入下的输出抖动，利于诊断可复现。

# 数据模型与知识结构

- `aiops.analysis_history`：核心字段 kind（告警类型）、target（告警内容）、result（结构化分析报告），保存每次诊断结果，供复盘与效果回看。
- `category_rocketmqcluster`（`RocketMQCluster` 模型，ticket 库只读）：RocketMQ 集群元数据（nameserver/owner/version/region/app_type），rocketmq 类告警分析时查集群信息用。
- 知识库「故障案例分析」（飞书表，`read_feishu_document` 读）：列含故障类型、故障现象、直接根因、解决方案、特别说明，是诊断的最高优先级依据。
- 节点池资源配置要求（飞书文档）：CPU/存核比等配置红线，OOM/CPU 类根因给解决方案时引用。
- 报告结构（强制模版）：故障根因（一句话「[原因]导致[结果]」或「未找到根因」）、分析凭据（客观现象+直接证据+解释）、解决方案（围绕案例方案，可扩展不可省略关键步骤）、参考链接（仅外部知识库链接，内部库不展示）。

# LLM 接入与演进

- 早期：`llms.py` 用 OpenAI SDK 直连三套——gpt-4o（`llm-proxy.qa-kubeflow`）、doubao-pro-32k、deepseek-r1（`test-doubao-llm-service-adapter`）。`single_agent.Player` 早期直接用 gpt-4o 跑 Function Calling。
- 现在：`TAOPlayer.create_completion` 统一走内部 AI Gateway（`prod-ai-gateway.soulapp-inc.cn`，endpoint `/llm/doubao-seed-2.0/...`，带 `soul-ai-service` 服务标识），自封装 `HttpRequestTool.post` 发请求，`_parse_gateway_response` 用 `SimpleNamespace` 把 dict 响应递归转成可点号访问、且补齐 `tool_calls/reasoning_content` 默认字段的 OpenAI 兼容对象。`create_completion_openai` 保留为备用 SDK 通道。
- 参数：`temperature=0`、`top_p=0.9`、`timeout=120`，诊断求确定性。
- 演进意义：把模型供给从「代码里散落的多 client」收口到统一 Gateway，便于切模型、计费、限流和审计，业务侧只依赖一个 OpenAI 兼容契约。

# 指标与收益

- 取证自动化：把容器告警的固定排查动作（看 Pod、查事件、拉指标、翻日志、看变更、看节点池）从人工手动收成 Agent 自动调用，降低重复排障成本。
- 经验结构化：把散落的排障经验沉淀成知识库故障案例，诊断有据可依，新人也能复用。
- 输出规范化：结构化报告 + 否决式质检，减少「可能/建议检查」这类不可执行结论，研发拿到就能动作。
- 风险可控：工具只读、报告不含命令行、匹配不上就「未找到根因」，把 LLM 用在运维场景的风险压到可接受。

不要在没有真实数据时说「诊断准确率 90%」「MTTR 下降多少」「自动处置了多少故障」。更稳妥的表达是：把固定排查动作自动化、把排障经验结构化、把 LLM 输出约束成证据驱动且受控。

# 和岗位的匹配点

- AIOps / 智能运维：告警根因分析 Agent、知识库故障案例、证据驱动诊断，可展开「智能运维不是聊天机器人，是受控的排障系统」。
- LLM Agent 工程：ReAct 引擎、Function Calling 工具设计、上下文管理、防幻觉、多 Agent 编排，可展开每种取舍。
- SRE / 稳定性平台：挂在告警链路下游、只读取证、链路隔离、失败不阻塞主通知，可展开稳定性设计。
- 可观测平台：Prometheus 多维指标、ClickHouse 日志、K8s 事件、变更记录的统一取证，可展开多源证据如何服务诊断。
- 平台后端：Django + DRF、RocketMQ、MySQL、AI Gateway 接入，可展开后端控制面与异步触发。

# 面试讲法

## 三十秒版

这是公司告警链路下游的根因分析 Agent，代号拉普拉斯。收到容器告警后，LLM Agent 按 ReAct 边推理边调只读运维工具取证——describe_pod、Prometheus 指标、K8s 事件、Pod 日志、变更记录、节点池状态，对照飞书知识库的故障案例逐项验证现象，匹配出唯一根因和解决方案，再过一套 12 项否决式质检，输出结构化报告回贴飞书。我主要参与诊断 Agent 核心链路的建设和改造，包括 ReAct 引擎、工具集、知识约束和多 Agent 编排。它比较能体现我把传统告警、K8s 排障和大模型 Agent 结合，并且懂得用工程手段约束模型、不让它幻觉。

## 三分钟版

背景是容器告警推给值班人后，查 Pod、看事件、拉指标、翻日志、对案例都靠人工，重复且慢；而直接把告警丢给大模型又会编数据、给不可执行的废话。这个系统的思路是把根因分析拆成可控环节：接入层从告警网关或 MQ 拿告警并解析服务树、Pod 名、时间；推理层用 `TAOPlayer` 跑 ReAct，think 决定调哪个工具、action 执行只读取证、observe 决定继续还是收口；知识层对照飞书「故障案例分析」逐项验证现象，形成「时间→现象→证据」链条，全部通过才认定根因，匹配不上就输出「未找到根因」；质检层用 `OBSERVER_PROMPT` 的 12 项否决清单兜底；输出层产结构化报告回贴飞书、落 `analysis_history`。

我在面试里会重点展开三块：第一是 ReAct 引擎怎么在 think/action/observe 间流转、怎么用轮数上限和强制汇总保证收敛；第二是工具为什么必须只读、结果怎么结构化回灌、空结果为什么要合法；第三是怎么用知识约束 + 逐项验证 + 否决式质检把幻觉空间压到最小。

## 五分钟版

更深入讲，这套系统的本质是「用工程手段把大模型约束成证据驱动、可控、可质检的排障系统」，而不是接个模型做问答。

它有两种形态：生产主力是单 Agent ReAct，`single_agent.start_analysis` 解析告警、预拉 Pod/服务上下文、读知识库案例、构造 prompt，然后 `TAOPlayer.think()` 跑 ReAct 循环。另一套是多 Agent 圆桌 `AsyncRoundTable`，把提因、规划、执行、裁决拆给 analyst/planner/k8s_engineer/examiner 五角色，各持一个 Queue 协作。

防幻觉是核心。第一层只读取证，所有工具是 K8s 资源/事件/日志、Prometheus 指标、变更、节点池这类只读接口，模型只能拿数据不能动线上。第二层知识约束，诊断对照飞书「故障案例分析」逐项验证故障现象，每项要有时间、现象、证据，全部通过才认根因，否则「未找到根因」，严禁推断。第三层否决式质检，终止前过 12 项检查，包括不确定表述、条件式结论、数据溯源、工具名替换、虚构路径、格式完整性，任一不过禁止调 terminate。第四层确定性，`temperature=0`，并把工具名和 kubectl 命令从报告里剔除，面向研发只给可执行动作。

我也会主动讲现状边界和改造点：当前 `max_tool_call_times=1` 容错偏弱；多 Agent 用线程加队列轮询，时序脆、难调试；知识库走飞书文档实时读取耦合较重；部分工具（memory、sniffer）偏占位；12 项质检全交给模型自检，纯规则能查的应该下沉成代码后置校验。如果让我重构，我会把编排换成明确的状态图/消息总线、案例迁到可检索知识库做 RAG 召回、给工具输出加结构化 schema、把确定性校验代码化。

```
                                                              └── Q12：现在哪里实现不合理？让你重构会怎么做？
```

# 高频 Q&A

## 这个项目和「接个大模型做告警问答」有什么区别？

区别在它不是开放式生成，而是受控的验证系统。工具只读取证、对照知识库案例逐项验证、强制数据溯源、否决式质检、单一根因兜底，每条结论都能追溯到工具原始输出，目标是把幻觉空间压到最小，而不是聊得像那么回事。

## 为什么用 ReAct 而不是固定流程或一次性生成？

因为容器告警的排查步数不固定，有的看 Pod 事件就够、有的要层层深挖。ReAct 让推理和取证交替进行，每步行动基于上一步证据；同时用轮数上限和强制汇总保证一定收敛。固定流程不够灵活，一次性生成又拿不到实时证据。

## 工具为什么必须只读？

运维场景里 Agent 自动执行写操作（删 Pod、改配置、重启）风险极高。只读保证它只能取证、给人建议，处置决策权留在人手里。这也是把 LLM 安全用在生产运维的前提。

## 怎么防止模型编数据？

四层：只读工具提供真实证据；prompt 强制每条凭据挂在具体指标/日志/事件上、禁止推测；对照知识库案例逐项验证现象；终止前 12 项质检查不确定表述、条件式结论、数据溯源。匹配不上知识库就输出「未找到根因」，不许硬编。

## 为什么宁可「未找到根因」也不让它猜？

运维里一个错误根因比没有根因更危险，会把人往错误方向带。保守输出加人工兜底，是比「强行给个看似合理的答案」更安全的设计。

## 单 Agent 和多 Agent 怎么选？

生产主力是单 Agent ReAct，因为容器告警诊断路径相对固定，单 Agent 足够且更稳、成本更低。多 Agent 圆桌把提因、规划、执行、裁决拆成专业角色，适合根因空间发散、要多专业并行验证的复杂场景，当前更像探索形态，并发还是手搓线程+队列，时序偏脆。

## 上下文会不会爆？

日志、事件、指标输出很长，当前用 `repr()[:N]` 截断、`_clean_messages` 规整消息、按需保留来缓解，多轮 + 轮数上限也限制了总量。这块是明显可改进点，更好的做法是把工具输出做成结构化证据对象，再按相关性做上下文裁剪和外置记忆。

## 12 项质检也是模型自检，怎么保证可信？

这是软约束，靠它兜语义类问题（逻辑一致、证据完整、单一根因）。纯规则能判的——是否含 kubectl 命令、是否含工具名、是否被 markdown 包裹、时间戳格式——应该用代码后置校验，代码里已有去 markdown 包裹的 `re.sub` 雏形，改造方向是把这类检查全部代码化，不依赖模型自觉。

## 知识库怎么接进来的？

通过 `read_feishu_document` 工具读飞书知识库的「故障案例分析」表，按告警类型筛出案例；工具输出里如果出现飞书链接，Agent 会自动抓取并查阅文档再继续推理，相当于一个轻量的工具触发检索。改造方向是迁到可检索知识库做向量召回。

## 为什么演进选 LangGraph 而不是继续用 LangChain？

我们当前只借了 LangChain 的 `StructuredTool` 定义工具，ReAct 循环是手搓的——因为 LangChain 的 `AgentExecutor` 把控制流封装成黑盒，循环不可控、没显式全局状态、不能 checkpoint 回放、多 Agent 也要自己搭，插质检 / 裁工具 / 改终止条件都得绕框架。但手搓又踩了「时序脆、难回放」的坑。LangGraph 把 Agent 建成有向状态图：think/act/observe 变成节点 + 条件边显式表达循环和分支，全局 `State` 显式共享，内置 checkpointer 支持回放和暂停恢复，原生 human-in-the-loop 给「受控处置」留审批位，多 Agent 也有 supervisor/network 拓扑。所以演进是把状态机 / 并发 / checkpoint 交给框架，而不是继续手搓或退回链式抽象。详见「编排框架选型」。

## 为什么从 OpenAI SDK 直连改成走 AI Gateway？

早期代码里散落 gpt-4o/doubao/deepseek 多个 client，切模型、计费、限流、审计都不好做。统一走内部 AI Gateway 后，业务侧只依赖一个 OpenAI 兼容契约，`_parse_gateway_response` 负责把 Gateway 响应转回可点号访问的兼容对象，模型供给和业务解耦。

## LLM 失败或工具失败了怎么办？

工具失败被 `execute_tool` 捕获并回灌成「工具调用失败」，observe 阶段在 `max_tool_call_times` 内重试；推理超轮数强制汇总兜底；整体诊断异常 `send_clerk` 告警但不阻塞主告警通知。当前容错偏弱（重试只 1 次），是改造点。

# 风险与边界

- 容易被追问的说法：「我做了一套公司级 AIOps 平台」。
  - 风险：会被追组织归属、规模、上线范围、效果数据。
  - 更稳妥的表达：我主要参与诊断 Agent 后端核心链路的建设和改造（ReAct 引擎、工具集、知识约束、多 Agent 编排）。
- 容易被追问的说法：「能自动处置线上故障」。
  - 风险：工具全是只读，根本没有处置能力。
  - 更稳妥的表达：系统只读取证、输出建议，处置决策权在人，这是有意的安全设计。
- 容易被追问的说法：「诊断准确率/MTTR 改善了多少」。
  - 风险：没有真实评测数据，会当场穿帮。
  - 更稳妥的表达：用定性收益（取证自动化、经验结构化、输出规范化、风险可控），不报没有数据支撑的指标。
- 容易被追问的说法：「多 Agent 是成熟的生产能力」。
  - 风险：并发是手搓线程+队列轮询，终止判定脆，更像探索形态。
  - 更稳妥的表达：生产主力是单 Agent ReAct，多 Agent 是探索方向，我能讲清它的取舍和重构思路。

# 评测与成功口径

agent 项目几乎必被问"你怎么衡量它好不好"。根因诊断没有唯一标准答案、生产样本长尾且无标注，所以评测难——但**能讲清「该怎么评」本身就是加分项**。诚实前置：目前还没有评测集和量化口径（见「当前设计的缺陷」），下面是方法论与规划，不是已有数据，别报准确率数字。

按四层拆评测维度：

- **结果层**：根因准确率（Top-1 / Top-k 与人工复盘结论比对）、误报率（瞎给根因）、漏报率（该找到没找到）、以及「未找到根因」判得是否恰当——这一项对运维尤其重要，宁可漏报不可误导。
- **过程层**：关键证据取证完整性、收敛轮数、工具调用成功率、结论是否都有数据溯源。
- **工程层**：端到端延迟、单次 Token 成本、超时/失败率。
- **安全层**：幻觉率（结论无证据支撑）、是否守住只读边界。

怎么把它落地（演进方向）：

- **离线 golden set**：用历史告警 + 人工复盘结论构造标注集，每次改 prompt / 工具 / 范式都回放跑分，防回退。
- **LLM-as-judge + 人工抽检**：开放式根因用裁判模型按上面维度打分，人工校准裁判。
- **在线弱信号**：影子运行（shadow）对比人工诊断、运维对建议的**采纳率**作为效果代理指标。

> 面试防守：被追"现在准确率多少"——直接说"还没有量化评测口径，这正是我列为缺陷、要在演进里补的；但我能讲清评测该怎么设计"。比硬编一个数字安全得多。

# 当前设计的缺陷

站在现在看这套设计哪里不好——有些还没出事，但属于判断得到的不足。只点问题，解法见「后续演进」。

- **范式错配：ReAct 直接跳多 Agent，跳过了 Plan-Execute**。容器告警诊断路径相对固定、根因空间收敛，本不需要多专家发散验证；先上最重的多 Agent，等于没拿到「显式规划」的收益就背上了全部编排成本。（→ 见后续演进中期）
- **编排是手写递归 + 手搓线程队列，没有状态图框架**。`think/act/observe` 靠互相递归驱动、多 Agent 靠线程+队列轮询，导致时序脆、难回放、难加节点。（→ 见后续演进中期 LangGraph）
- **质检是软约束，依赖模型自检**。终止前 12 项检查里，连「是否含 kubectl 命令、是否被 markdown 包裹、时间戳格式」这种纯规则项都还部分交给模型判断，可靠性不足。（→ 见后续演进短期）
- **上下文管理粗暴，不可扩展到长链路诊断**。`repr(result)[:N]` 一刀切截断、无结构化证据对象、无外置记忆，链路一长就失忆或顶爆。（→ 见踩坑 1）
- **容错薄**。`max_tool_call_times=1`，工具失败只重试一次、不区分错误类型、无退避。
- **知识库靠手维护、召回不可控**。故障案例是飞书文档表 + tool 触发抓链接，没有系统化向量召回，案例也不会自动沉淀。
- **诊断质量不可量化**。没有评测集和成功口径，"这次诊断到底准不准"只能靠人事后判断——这是这类 AIOps 系统最硬的短板。（→ 见评测与成功口径）

# 上线后踩过的坑

挑三个最能复盘、也最常被追问的。每个都写「现象→根因→当时怎么兜→真正的解」，不回避兜法的不完美。

## 上下文膨胀，关键证据被截

- 现象：日志 / 事件 / 指标输出极长，推理跑几轮后模型开始「失忆」——忘掉前面取到的证据、重复取证，偶发请求超长报错，Token 成本也偏高。
- 根因：工具结果直接进对话历史，早期用 `repr(result)[:N]` 一刀切截断，要么把关键证据截没了、要么截完仍把上下文顶大。
- 当时怎么兜的：加 `_clean_messages` 规整消息结构、按需保留，再靠 `max_dialog_rounds` 轮数上限压总量——是缓解，不是根治。
- 真正的解：把工具输出做成结构化「证据对象」，长内容外置存储、上下文只留摘要 + 引用，按相关性裁剪（见后续演进中期项）。

## 模型「硬凑」一个看似合理的根因

- 现象：匹配不到已知故障的告警，模型仍倾向给出一个像模像样的根因——而运维里一个错误根因比没有根因更危险，会把人带偏。
- 根因：生成式模型天然倾向「给答案」，光靠 prompt 说「不许猜」压不住。
- 当时怎么兜的：加了强制数据溯源 + 终止前的否决式质检；匹配不上知识库就强制输出「未找到根因」，不许编。
- 真正的解：质检里纯规则能判的（是否含 kubectl 命令、工具名、markdown 包裹、时间戳格式）从「模型自检」迁到代码后置校验，模型只留语义类判断——别让兜底也依赖模型自觉。

> 诚实边界：知识约束与质检这块我参与了设计与改造；「未找到根因」兜底是被这个坑逼出来的，不是一开始就有。

## 多模型供给混乱，切换/计费/限流都难

- 现象：早期代码里散落 gpt-4o / doubao / deepseek 多个 client，换模型要改业务代码，计费、限流、审计都没法统一做。
- 根因：业务直连各家 SDK，模型供给和业务逻辑耦合。
- 当时怎么兜的：先收敛 client 创建入口，减少散落。
- 真正的解：统一走内部 AI Gateway，业务侧只依赖一个 OpenAI 兼容契约，`_parse_gateway_response` 把 Gateway 响应转回兼容对象，模型供给与业务彻底解耦。

# 后续演进

主线：**从「手写 ReAct 单 Agent」演进成「Plan-Execute 为核心、按需多 Agent、可观测可处置」的诊断系统**。分层推进，诚实区分已做 / 在做 / 想做。

- **短期 · 工程化止血（部分在做）**：质检规则类检查代码化后置校验（已起步，`re.sub` 去 markdown 包裹是雏形）；`max_tool_call_times` 改成带退避、区分错误类型的有限重试；按告警类型动态裁剪工具集、缩小 tools schema；给 Agent 自身上 trace（每步推理 / 工具调用 / Token / 耗时 / 是否命中知识库）= AgentOps，正好接「Agent 可观测」。
- **中期 · 范式升级（想做，核心改造）**：把执行范式从 ReAct 升到 **Plan-and-Execute**——先产出可审计的诊断计划，独立证据步并行采集（降延迟、降成本），失败 / 新证据触发局部 replan；并用 **LangGraph 状态图**替掉手搓的 think/act/observe 递归和多 Agent 的线程+队列轮询，把状态机 / 并发 / checkpoint / 回放交给框架，直接消掉「时序脆」这个坑。配套把故障案例从飞书文档迁到可检索向量库做相似召回。
- **长期 · 选择性多 Agent + 闭环（探索）**：多 Agent 圆桌只对发散型复杂故障（跨服务雪崩、多因耦合）启用，且每个 Agent 内部是 Plan-Execute、编排在 LangGraph 上，而不是对所有告警都圆桌；诊断结果自动回流沉淀成新案例形成知识闭环；在严格审批和分级动作门控下，谨慎从「只读诊断」探索到「受控处置」（一号响应者）。

> 为什么不直接上多 Agent：容器告警诊断路径相对固定、根因空间收敛，Plan-Execute 单 Agent 的 ROI 远高于多 Agent 的编排成本——当前的多 Agent 圆桌其实是超前设计，演进上我会先退回把 Plan-Execute 地基打实。
>
> 注意：处置自动化、效果指标这些是规划方向，不要讲成已经上线。

# 编排框架选型：为什么是 LangGraph 而不是 LangChain

这一节是「后续演进」里 LangGraph 改造的展开论证，回答面试常问的「为什么不用 LangChain」。诚实前置：我们当前**没有用 LangChain 的 AgentExecutor**，只借了它的 `StructuredTool` 做工具定义，ReAct 循环和多 Agent 是手搓的——这恰恰是因为 LangChain 的链式抽象不适合排障 Agent，而手搓又踩出了「时序脆、难回放」的坑，所以演进选型落在 LangGraph。

## LangChain 这类链式框架的局限

- **控制流是黑盒，难精细介入**：`Chain` / `AgentExecutor` 把「调模型→解析→调工具→再调模型」的循环封装在框架内部，想在某一步前插质检、按告警类型裁剪工具、改终止条件，都要绕框架或 monkey-patch。排障 Agent 恰恰需要在 think/action/observe 每个节点插自定义逻辑（轮数上限、强制汇总、否决式质检），链式抽象反而碍事——这也是我们当初宁可手搓的原因。
- **控制流是固定的,放不下带状态的循环**：LangChain 早期是线性 Chain(输入 → PromptTemplate → LLM → OutputParser → 输出),干净但控制流写死。一旦要「LLM 觉得要查库就查、不查就跳过」或「失败重试三次」,就只能把判断硬编码进节点或在外层 if/else 手动管流程;而排障是「条件循环 + 分支」——现象验证失败要排除案例继续下一个、工具失败要重试、证据够了才收口,链式管道很难显式表达。
- **状态管理是隐式的**：链之间靠输入输出传递，缺一个**显式、可读写、可持久化的全局状态**；我们现在靠 messages 列表 + 一堆布尔标志（`reason_verify_completed` / `step_verify_completed`）手维护状态，正是这个缺口的体现。
- **无原生持久化 / 回放 / 暂停恢复**：链跑挂了就从头来，没有 checkpoint，不能 time-travel 回放某一步，也很难做 human-in-the-loop 审批；这对要「受控处置」的运维场景是硬伤。
- **多 Agent 要自己搭**：LangChain 没给多 Agent 的一等编排，我们的圆桌只能用「线程 + asyncio.Queue + 轮询 sleep」手搓，时序脆、难调试。

## LangGraph 的状态图模型怎么对症

LangGraph 的本质是**把 LLM 变成一台状态机**，靠 `StateGraph` 三要素：`State`（数据快照=对话历史+工具结果+中间变量）、`Node`（执行动作并产出状态更新=调模型/调工具/业务逻辑）、`Edge`（决定下一步,普通跳转或**条件边**）。`compile()` 把「声明式的图」编译成「可执行的调度器」,调度器本质是一个事件循环:执行节点 → Reducer 合并状态 → 查出边(普通边直达,条件边调路由函数按运行时 `State` 决定去向) → 入队 → 循环到 `END`。对应解掉上面每一条:

- **think / action / observe → 节点 + 条件边的循环图**。文章点透一句:「ReAct Agent 的本质就是一个带条件边的循环图」——`llm` 节点产出 tool_calls 就路由到 `tools` 节点、执行完回到 `llm`,不需要了就到 `END`。这正是我们手搓递归在做的事,用 LangGraph 后循环和分支变成图里显式、可命名、可单测、可插桩的边。
- **布尔标志满天飞 → 显式 `State` + Reducer**。节点返回的不是「新 State」而是「更新片段」,由每个字段的 Reducer(追加 / 覆盖 / 累加)合并;`reason_verify_completed` 这类标志收进 `State` 由 Reducer 维护,谁读谁写一目了然。
- **「时序脆、难回放」→ 不可变快照 + checkpointer**。`State` 每步产生新快照、旧快照保留(不可变),这正是 **Checkpoint 的底层基础**;配 checkpointer 持久化后支持暂停 / 恢复、断点续跑、time-travel 回放,这个坑直接从机制上消掉。
- **并行取证 / 多 Agent → Fan-out / Fan-in**。一个节点连多个后继即并行(`search_web` / `search_db` 同时入队),merge 节点等所有前驱就绪再汇聚——这正是 Plan-Execute「独立证据步并行采集(降延迟降成本)」和多 Agent 协作的框架底座,不用再手搓线程+队列轮询。
- **Agent 可观测 → `stream()` 逐节点 yield**。每个节点完成就 yield 一次 `{节点名: 状态更新}`,天然是 AgentOps 的 trace 点(每步推理 / 工具调用 / 是否命中知识库),对上短期演进的「给 Agent 上 trace」。
- **受控处置 → human-in-the-loop**。在图里插入断点等人审批,给「从只读诊断走向受控处置」留出标准门控位置。

## 落到本项目的收益

| 痛点（现状手搓） | LangChain 为什么不解 | LangGraph 怎么解 |
| --- | --- | --- |
| think/act/observe 靠递归驱动 | 线性 Chain 控制流固定,放不下循环 | 节点 + 条件边的循环图(ReAct 本质),可单测可插桩 |
| 一堆布尔标志手维护状态 | 状态隐式、靠链传递 | 显式 `State` + Reducer(更新片段合并) |
| 跑挂从头来、无法回放调试 | 无 checkpoint | 不可变快照 + checkpointer,time-travel 回放 |
| 并行取证 / 多 Agent 线程轮询、时序脆 | 无并行模型、无一等多 Agent | Fan-out / Fan-in 并行,merge 自动汇聚 |
| Agent 没 trace、调试靠打印 | 黑盒,中间步看不见 | `stream()` 逐节点 yield 状态更新 |
| 处置自动化没有审批位 | 无原生 human-in-the-loop | 图内断点人审门控 |

面试一句话收口：我们手搓 ReAct 不是因为不知道有框架，而是 LangChain 的链式抽象对「条件循环 + 显式状态 + 可回放 + 人审门控」这些排障刚需支持不好；踩了「时序脆、难回放」的坑后，演进选型自然落到把状态机 / 并发 / checkpoint / 回放交给 LangGraph，而不是继续手搓或退回 LangChain。
