# 项目定位
+ 项目名称：SAE CICD 能力 Agent 化（Agentic DevOps 的受控工具层）
+ 所属领域：云原生 / 平台工程 / CICD / AI 工程化（Agent + MCP）
+ 项目类型：在已有 CICD 控制面（SAE）上，把发版、查询、回滚、Pod 排障等运维能力封装成受控的 MCP Skill，接入内部 Agent
+ 核心目标：让 Agent 能像研发一样安全地调用平台能力，把高频运维操作从"人工点控制台"变成"Agent 可编排的受控工具"，同时不丢失权限、审计和操作边界
+ 服务对象：内部研发、SRE、以及内部 Agent 平台（Moss-Compose）/ Coding Agent
+ 关键词：MCP、Agent-Ready、受控自治、身份绑定、权限审计、Agentic DevOps、CICD 控制面
+ 面试价值：这是少见的"平台视角"的 AI 工程化实践——不是用 Claude Code 写业务代码，而是把运维平台本身改造成 Agent 可安全操作的对象。同时命中通义"Agent Infra / Coding Agent 辅助"和阿里云"Agent + AI Coding + 运维自动化平台化"两类岗位

边界先讲清：这是真实做过的能力（SAE `/mcp/v1` 工具层 + 鉴权边界），不是理论。但它处在 Agent 化演进的 **L2「能力 Agent 化」地基层**，不是"已经做出了 Agent 全自治的平台"——面试时我会主动点明这条边界，反而显得想得清楚。

术语口径：这套"让 Agent 受控操作交付平台"的实践，业界多称 Agentic DevOps / 平台能力 Agent 化；它和 AIOps（观测、告警、根因分析）不是同一回事，别混。"AI Native Ops" 不是公认术语，面试里别硬用——要么说 Agentic DevOps，要么直接讲能力本身。

# 项目背景
+ 背景问题：研发日常交付要在应用、构建、镜像、发布、回滚、日志、事件、Pod 排障多个系统之间来回拼操作。
    - 具体表现：一次"发版后发现异常要回滚"的动作，人要在控制台里查应用、看 Pod、看 Rollout 状态、点中止——链路长、上下文切换多。
    - 问题方向：当内部开始建 Agent 平台（Moss-Compose）、研发也在用 Coding Agent 时，自然的诉求是"能不能让 Agent 直接帮我查状态、触发发版、回滚灰度"。
+ 但直接把能力开放给 Agent 有风险：
    - 具体表现：如果给 Agent 一个绕过权限的"超级接口"，等于开了一个不受控的特权后门——谁都能借 Agent 之名动生产，出了事查不到人、拦不住越权。
    - 问题方向：所以真正要解决的不是"让 Agent 能调"，而是"让 Agent 在和人类同一套权限 / 审计 / 边界下安全地调"。
+ 不做的风险：要么 Agent 接入停留在"只读问答"没有真正价值，要么为了能力开放牺牲了平台的治理底线。

# 项目目标
+ 能力工具化：把 SAE 已有的发版、查询、回滚、Pod 排障等能力，封装成 Agent 可调用的标准 MCP 工具。
    - 说明：让 Agent 拿到的是"语义化的运维动作"，而不是 Kubernetes / Tekton / Rollouts 的原生复杂度。
+ 受控自治（核心）：Agent 调用复用人类同一套身份、权限、审计链路，不开特权后门。
    - 说明：Agent 调用要能回答"是谁、经由 Agent、对哪个应用做了什么"，越权要被同一套规则拦住。
+ 平台不另起炉灶：MCP 入口复用控制台同一套 Controller，避免维护两套逻辑、两套权限。
    - 说明：降低维护成本，也避免"控制台拦得住、MCP 拦不住"的治理漏洞。
+ 为 Agent 化演进铺地基：先把"能力 Agent 化"这层做扎实，给后续 Agent 自主编排发布 / 排障闭环留出受控接口。

# 我的职责
+ 负责 SAE 后端控制面的 MCP / Agent 入口能力建设：
    - 具体工作：在 `sae-console` 注册 `/mcp/v1` 路由组，把 application / pipeline / rollout 等域的高频运维操作暴露成 MCP 工具。
    - 面试可展开点：为什么选择"复用 Controller"而不是"新写一套 Agent 专用接口"。
+ 负责 Agent 入口的鉴权与边界设计：
    - 具体工作：用独立的 MCP 鉴权中间件把 Agent 调用绑定到真实用户身份，并叠加平台资源 / 权限中间件，让 MCP 调用走和人类一致的权限校验。
    - 面试可展开点："受控自治"如何落地，越权和审计怎么保证。
+ 参与与内部 Agent 平台（Moss-Compose）的对接：
    - 具体工作：以 MCP 协议提供工具清单（工具名、参数、入参位置、请求模板），让 Agent 端按标准发现和调用。

职责口径：我做的是平台控制面的工具层和鉴权边界，不是自研 MCP 协议、不是自研 Kubernetes / Tekton / Rollouts，也不是已经做出了 Agent 自主决策闭环。

# 总体架构

## 架构分层
+ 入口层（双入口，单治理）：
    - 人类入口：控制台 / OpenAPI，走 CAS 鉴权 + 权限校验 + 审计。
    - Agent 入口：`/mcp/v1`，走 MCP 鉴权中间件（加密 token 绑定真实用户）+ 资源 / 权限中间件。
    - 关键点：两个入口在 `sae-console` 里收敛到同一层，再往下复用同一套业务逻辑。
+ Controller 层（复用，不另起后门）：
    - application Controller：list / queryAppInfo / getPods / batchUpdateImage。
    - pipeline Controller：trigger / check / getCommit。
    - rollout Controller：getRolloutInfo / promoteRollout / abortRollout。
    - 作用：人类点控制台和 Agent 调 MCP，最终落到的是同一套 Controller、同一套权限、同一套审计。
+ 底座层（复用，不自研）：
    - Kubernetes / Tekton 构建 / Argo（内部 Rollouts）/ GitLab / Harbor。
    - 作用：MCP 工具最终的真实执行落在这些底座上，平台只做语义收敛和治理。

架构一句话：把"Agent 能调平台"实现成"Agent 走和人类一样的门，进同一个治理后端"，而不是给 Agent 单独开一扇没锁的后门。

# 核心能力
+ 受控 MCP 工具集：把高频运维动作暴露成标准工具。
    - 说明：覆盖查询（应用列表 / 应用详情 / Pod / 流水线状态 / Rollout 状态 / git commit）、发版（触发流水线、批量更新镜像）、灰度控制（promote 继续 / abort 回滚）、排障（查 Pod）。
    - 面试展开点：工具粒度怎么定——按"语义化运维动作"而不是"裸 API"，让 Agent 调用结果可预期。
+ 身份绑定鉴权：把 Agent 调用还原成"某个真实用户在操作"。
    - 说明：MCP 鉴权中间件校验调用方携带的加密 token，解出真实用户标识并写入上下文，后续权限和审计都基于这个真实身份。
    - 面试展开点：为什么 Agent 不能是"匿名超级权限"，而必须绑定到人。
+ 权限 / 资源复用：MCP 调用叠加平台资源中间件，复用既有权限模型。
    - 说明：该用户对该应用 / 集群有没有操作权限，由和控制台一致的规则判定，越权同样被拦。
+ 失败可解释：执行失败返回可理解的原因回写给 Agent。
    - 说明：不静默吞错，Agent 才能据此决定下一步（重试、升级、告知用户）。
+ 工具自描述：以 MCP 协议提供工具名、描述、参数、入参位置和请求模板。
    - 说明：Agent 端能自动发现工具能力，降低对接成本。

# 核心流程
<img src="https://cdn.nlark.com/yuque/0/2026/png/29786964/1780472894738-aa94f1e1-a0d2-4763-9f74-b9e492cc8123.png" width="960" title="" crop="0,0,1,1" id="u3feb39d3" class="ne-image">

## 流程说明（以 Agent 发起「灰度回滚」为例）
+ 阶段一：意图 → 工具调用。
    - 关键动作：Agent 把"回滚某应用灰度"翻译成 MCP tool call（rollout/abortRollout）。
    - 相关组件：Agent（MCP Client）、MCP 工具清单。
+ 阶段二：身份绑定。
    - 关键动作：MCP 鉴权中间件校验加密 token，解出真实用户标识并写入上下文。
    - 失败处理：token 缺失 / 非法 → 401 拦截，Agent 拿不到任何能力。
+ 阶段三：权限 / 资源校验。
    - 关键动作：资源中间件按真实用户判定其对该应用 / 集群是否有操作权限。
    - 失败处理：无权限 → 拒绝执行，和人类越权同一套拦截。
+ 阶段四：复用 Controller 执行。
    - 关键动作：调用 rollout Controller 的 AbortRollout，落到底层 Argo / 内部 Rollouts。
    - 失败处理：底层异常 → 返回可解释失败原因，不静默吞。
+ 阶段五：审计落账。
    - 关键动作：记录"谁（真实用户）经由 Agent，对哪个应用做了回滚"。
+ 阶段六：结果回写 Agent。
    - 关键动作：返回执行结果 / 当前 Rollout 状态，供 Agent 继续编排或反馈用户。

流程要点：每一步都受控，非快乐路径（身份非法 / 越权 / 执行失败）都有明确处理，这正是"敢让 Agent 动生产"的关键。

# 关键设计
## 设计一：复用同一套 Controller，而不是给 Agent 新开接口
解决的问题：避免"控制台一套逻辑、Agent 一套逻辑"导致权限和行为不一致，甚至出现 Agent 绕过治理的后门。

方案设计：`/mcp/v1` 路由组复用 application / pipeline / rollout 的同一批 Controller 方法，MCP 入口只在前面加鉴权和资源中间件，业务执行完全复用。

为什么这样做：一套业务逻辑意味着一套权限、一套审计、一套行为语义；控制台能拦的，MCP 也一定拦得住。

取舍：MCP 工具的入参形态要去适配 Controller 既有签名，灵活性略受限；但换来的是治理一致性，对生产平台这笔账划算。

面试追问：如果 Agent 需要一个控制台没有的组合动作怎么办？答：优先在 Controller 层做编排或新增受控方法，仍走同一套鉴权 / 审计，而不是在 MCP 侧旁路实现。

## 设计二：把 Agent 调用绑定到真实用户身份
解决的问题：Agent 不能是"匿名的超级权限"，否则审计追不到人、越权拦不住。

方案设计：独立的 MCP 鉴权中间件，要求调用携带标识真实用户的加密 token，校验通过后把真实用户写入请求上下文，后续权限和审计都基于该身份。

为什么这样做：让"Agent 操作"在治理视角上等价于"某个真实用户操作"，责任和权限都落到人。

取舍：调用方（Agent 平台）要负责正确携带用户身份凭证，对接稍重；但这是受控自治不可省的一步。

面试追问：token 机制的安全性？答：这是身份绑定的工程实现，核心是"把 Agent 调用还原成真实用户 + 复用平台权限"；具体加密细节属于内部实现，我不会在面试里展开密钥之类的敏感信息，重点讲设计意图和边界保证。

## 设计三：工具粒度按"语义化运维动作"切
解决的问题：如果把裸 Kubernetes / Tekton API 直接丢给 Agent，调用结果不可预期、风险面大。

方案设计：工具按平台已有的运维语义暴露（查应用 / 查 Pod / 触发流水线 / 查 Rollout / promote / abort），每个工具有明确描述和参数。

为什么这样做：Agent 拿到的是"可解释、结果可预期"的动作，平台也能在动作粒度上做权限和审计。

取舍：工具集要随平台能力演进维护；但比起开放底层 API，可控性高很多。

# 技术难点
+ 难点：能力开放与治理底线的平衡。
    - 为什么难：开放能力越多越方便 Agent，但每开放一个写操作（发版 / 回滚）都是生产风险面。
    - 解决思路：复用同一套权限 / 审计 + 身份绑定 + 按语义动作切粒度，让开放是"受控开放"。
    - 面试展开点：哪些动作适合先开放（查询、灰度 promote/abort 这类可控动作），哪些要更谨慎。
+ 难点：双入口的一致性。
    - 为什么难：人类入口和 Agent 入口很容易演化成两套逻辑，留下治理缝隙。
    - 解决思路：MCP 入口只加中间件、不改业务，强制复用 Controller。
    - 面试展开点：如何保证"控制台能拦的 MCP 也能拦"。
+ 难点：失败语义对 Agent 友好。
    - 为什么难：Agent 依赖返回结果决策，静默失败或模糊错误会让它做出错误的下一步。
    - 解决思路：失败原因可解释回写，区分鉴权失败 / 越权 / 执行失败。

# 稳定性与治理
+ 权限 / 审计：MCP 调用复用平台权限模型，所有动作绑定真实用户、可审计追溯。
    - 作用：Agent 操作可追责、越权可拦截。
+ 失败隔离与可解释：鉴权失败、越权、执行失败分别返回明确语义，不静默。
    - 作用：Agent 能据此正确决策，运维能据此排查。
+ 入口一致性：Agent 入口不旁路业务逻辑，复用同一套 Controller。
    - 作用：避免治理缝隙。
+ 动作分级（演进方向）：当前以查询 + 可控灰度动作（promote/abort）和镜像更新为主，对更高风险动作保持谨慎，逐步引入审批。
    - 作用：受控开放，控制生产风险面。

# 指标与收益
+ 维度：运维链路收敛
    - 结果：发版 / 查询 / 回滚 / Pod 排障从"多系统手工拼操作"变成"Agent 可编排的语义动作"。
    - 说明：定性收益，降低上下文切换和接入成本，未做严格量化口径。
+ 维度：治理不打折
    - 结果：Agent 调用与人类调用走同一套权限 / 审计，没有为了接入牺牲治理。
    - 说明：这是这套设计最核心的价值点。
+ 维度：可演进性
    - 结果：把"能力 Agent 化"做成地基，为后续 Agent 自主编排闭环（L3）留出受控接口。

口径说明：本项目收益以"能力可被 Agent 安全调用、治理一致"为主，不编造效率提升百分比；规模 / 收益数字若被追问，我会说"以定性为主，没有严格 A/B 量化"。

# 面试讲法
## 三十秒版
我在 SAE（公司内部 CICD 控制面）做了一件事：把发版、查询、回滚、Pod 排障这些运维能力，封装成 Agent 可调用的 MCP 工具，接进内部 Agent 平台。难点不是让 Agent 能调，而是让它在和人类同一套身份、权限、审计下安全地调——Agent 入口复用控制台同一套 Controller，调用绑定到真实用户，越权同样拦、动作同样审计。我把它定位成 CICD 平台能力 Agent 化的地基层。

## 三分钟版
在三十秒版基础上补：背景是内部建 Agent 平台、研发用 Coding Agent，自然想让 Agent 帮忙做运维，但直接开放能力等于开特权后门。我的做法是在 `sae-console` 注册 `/mcp/v1`，复用 application / pipeline / rollout 的同一套 Controller，前面加 MCP 鉴权中间件（把调用绑定到真实用户）和资源权限中间件。以"灰度回滚"为例：Agent 发起 abortRollout → 校验身份 → 校验权限 → 复用 Controller 执行 → 审计落账 → 结果回写，身份非法 / 越权 / 执行失败都有明确处理。关键设计是"复用 Controller 而不是新开接口"，保证控制台能拦的 MCP 也拦得住。

## 五分钟版
在三分钟版基础上补取舍和演进：取舍上，复用 Controller 牺牲了一点工具入参的灵活性，换来治理一致性，对生产平台划算；工具按语义动作切粒度而不是开放裸 API，控制风险面。边界上我会主动说清——我做的是 Agent 化演进的 L2"能力 Agent 化"地基，不是已经做出了 Agent 自主决策闭环（L3）。但 L2 的受控边界恰恰是 L3 敢落地的前提：Agentic DevOps 真正难的不是让 Agent 调工具，而是敢让它调——出错谁兜底、越权怎么拦、动作能否审计回放，这些我是真做了的。后续演进方向是动作分级 + 审批，再到 Agent 自主编排"发布→观测→异常→回滚"闭环。

# Agent 化演进定位（L1 / L2 / L3）
<img src="https://cdn.nlark.com/yuque/0/2026/png/29786964/1780472804294-40cf9812-811d-42d7-92fa-a1eb426b826d.png" width="831" title="" crop="0,0,1,1" id="u75dc5718" class="ne-image">

+ L1 AI 辅助：人用 Claude Code / Copilot 写代码、pipeline——普遍，不差异化。
+ L2 能力 Agent 化（Agent-Ready）：运维能力封装成受控 MCP Skill，身份绑定 + 权限 + 审计——**我真做的在这层**。
+ L3 自主编排 / Agentic DevOps：Agent 作为一等操作者自主编排闭环，平台围绕 Agent 交互设计——目标，能讲演进，未全做到。

面试一句话：我做的是 L2 地基；L2 的受控边界，正是 L3 自治敢落地的前提。这样讲既体现了对 Agent 化演进的判断，又不会被"那 Agent 自主决策了吗"一句戳穿。

# 高频 Q&A
## 这个项目到底解决了什么
让 Agent 能安全地操作 CICD 平台。把发版、查询、回滚、Pod 排障封装成 MCP 工具接进 Agent，同时保证 Agent 调用和人类走同一套身份、权限、审计，不开特权后门。

## 为什么不直接给 Agent 写一个超级接口
那等于开了个不受控的后门——谁都能借 Agent 之名动生产，出事追不到人、越权拦不住。平台的治理底线不能为了接入方便而打折，所以我让 Agent 入口复用人类同一套 Controller 和权限审计。

## 这算 AI Native 吗
算地基层，但我不会说"我做了 AI Native 平台"。这套实践更准确的叫法是 Agentic DevOps / 平台能力 Agent 化，注意别和 AIOps（观测、根因）混。AI 进交付运维有三个台阶：L1 人用 Coding Agent 辅助、L2 能力 Agent 化、L3 Agent 自主编排闭环。我做的是 L2——把运维能力做成受控 Skill。L3 那种 Agent 自主决策我没全做到，但 L2 的受控边界正是 L3 敢落地的前提。

## 受控自治具体怎么落地
三件事：一是 Agent 调用绑定真实用户身份，操作可追责；二是复用平台权限模型，越权和人类一样被拦；三是动作审计落账，谁经由 Agent 做了什么可查。这样 Agent 操作在治理上等价于某个真实用户操作。

## 复用 Controller 有什么好处和代价
好处是一套业务逻辑就有一套权限、一套审计、一套行为语义，控制台能拦的 MCP 一定拦得住，不留治理缝隙；维护成本也低。代价是 MCP 工具入参要适配 Controller 既有签名，灵活性略受限，但对生产平台这笔账划算。

## 工具粒度怎么定的
按"语义化运维动作"切，而不是开放裸 Kubernetes / Tekton API。Agent 拿到的是"查 Pod""触发流水线""回滚灰度"这种结果可预期的动作，平台也能在动作粒度上做权限和审计，风险面比开放底层 API 小很多。

## Agent 调用失败了怎么办
失败要可解释。鉴权失败返回 401、越权返回拒绝、执行失败返回可理解的原因，不静默吞错。因为 Agent 依赖返回结果决定下一步，模糊错误会让它做出错误动作。

## 哪些动作敢开放给 Agent，哪些要谨慎
查询类（应用 / Pod / 流水线 / Rollout 状态）最安全可以放开；灰度的 promote / abort 这类可控、可逆动作也相对安全；更高风险的动作要更谨慎，演进方向是引入动作分级和审批，而不是一次性全开。

## 和 AIOps 的受控处置是一个理念吗
是。AIOps 里我也讲"诊断可自动、高风险动作要审批受控"，和这里"能力受控开放、动作分级"是同一套思路——都是 Agent 可以做事，但必须在权限、审计、边界之内。

## 你在这个项目里具体写了什么
我负责 SAE 后端的 MCP / Agent 入口：注册 `/mcp/v1` 路由、把 application / pipeline / rollout 的运维操作暴露成 MCP 工具、设计 MCP 鉴权中间件把调用绑定到真实用户、叠加资源权限中间件，并以 MCP 协议提供工具自描述对接 Agent 平台。我不自研 MCP 协议、不自研 Kubernetes / Tekton / Rollouts。

## 如果继续做你会怎么演进
短期：把动作分级和审批补上，对高风险写操作加确认环节；把更多排障能力（日志、事件、诊断）做成工具。长期：从"Agent 调单个工具"走到"Agent 自主编排闭环"——比如发版后自动观测、发现异常自动建议或执行回滚，但始终在受控边界内。

# 风险与边界
+ 容易被追问的说法：把它讲成"我做了 AI Native CICD 平台"。
    - 风险：会被追问"Agent 自主决策了吗、闭环了吗"，答不上就穿帮。
    - 更稳妥的表达：我做的是"能力 Agent 化"的 L2 地基层，受控边界是 L3 自治的前提。
+ 容易被追问的说法：夸大 Agent 已经在生产自动发版 / 自动回滚。
    - 风险：没有自主闭环的证据。
    - 更稳妥的表达：当前是 Agent 在人的编排下调用受控工具，自主闭环是演进方向。
+ 容易被追问的说法：讲成自研了 MCP 协议或底层系统。
    - 风险：MCP 是开放协议，底座是开源 / 内部已有。
    - 更稳妥的表达：我做的是工具层封装、鉴权边界和平台对接，复用协议和底座。
+ 容易被追问的说法：在面试里展开 token 加密细节 / 密钥。
    - 风险：泄露内部实现 / 敏感信息，且不是重点。
    - 更稳妥的表达：只讲"身份绑定 + 复用权限审计"的设计意图，不展开密钥实现。

# 后续演进
+ 方向：动作分级与审批
    - 短期可落地：对发版、回滚等写操作加分级和确认 / 审批环节，进一步控制风险面。
    - 长期演进：形成"低风险自动、高风险审批"的统一受控策略。
+ 方向：排障能力工具化
    - 短期可落地：把日志、事件、诊断等更多排障能力做成 MCP 工具，让 Agent 能完整取证。
    - 长期演进：Agent 自主完成"发现异常 → 取证 → 给结论 / 建议"的排障链路。
+ 方向：从工具调用走向自主编排（L2 → L3）
    - 短期可落地：固化几条高频运维 SOP，让 Agent 按 SOP 编排多个工具。
    - 长期演进：Agent 作为一等操作者自主编排"发布 → 观测 → 异常 → 回滚"闭环，平台围绕 Agent 交互设计——始终在受控边界内。
